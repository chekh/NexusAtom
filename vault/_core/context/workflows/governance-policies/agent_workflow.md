# Правило: Рабочий процесс для AI-агентов

## Тезис
Агенты — полноправные участники, но с чёткими границами: они читают контекст, генерируют артефакты, валидируются ядром, но не обходят гейты и не пишут напрямую в ФС.

## Разрешённые действия агентов
| Действие | Команда / Механизм | Ограничения |
|----------|-------------------|-------------|
| Чтение контекста | `nexus context load --pack <role>` | Только узлы с `status: active/approved`, если не указано `--include-raw` |
| Захват идеи/заметки | `nexus capture --type note --content "..."` | Статус `raw`, требует человеческого `refine` для продвижения |
| Генерация черновика атома | `nexus atom create --type atom --from-research <id>` | Статус `draft`, требует `approve` перед стартом разработки |
| Обновление `tasks.md` | Прямая запись в `tasks.md` внутри атома | Только если атом в `in_progress`, manifest не меняется |
| Предложение изменений в `spec.md` | Создание `changes/` или `draft/` версии | Не меняет утверждённый контракт без `rfc` → `approved` перехода |

## Запрещённые действия
| Действие | Почему запрещено | Альтернатива |
|----------|-----------------|--------------|
| Прямая запись в `vault/` мимо CLI/API | Обходит валидацию, ломает `manifest`, создаёт битые ссылки | Использовать `nexus capture/create/update` |
| Изменение `status` на `approved` без человеческого аппрува | Критические решения требуют человеческого суждения | Оставить `draft`/`rfc`, отправить на ревью |
| Удаление файлов из `manifest.yaml` | Ломает трассировку, скрытые изменения | Использовать `nexus manifest update --remove <path>` с валидацией |
| Создание циклических зависимостей | Блокирует граф, ломает планирование | `nexus validate --check-cycles` перед коммитом |

## Протокол эскалации
```
1. Агент генерирует артефакт → status: raw/draft
2. Если тип критический (adr, governance, architecture) → требует человеческого аппрува
3. Человек ревьюит → `nexus approve <id>` или `nexus request-changes <id> --reason "..."`
4. При аппруве → статус меняется, узел попадает в активный граф
5. При отклонении → агент получает фидбек, может перегенерировать
```

## Интеграция с ядром
- **Валидатор**: Проверяет каждый артефакт против `registry.yaml` (обязательные поля, допустимые переходы FSM).
- **Манифест-трекер**: Фиксирует `content_hash` при создании, детектирует несанкционированные изменения.
- **Граф-резолвер**: Автоматически строит связи по `traces.*` и `[[id]]`, детектирует циклы.
- **Аудит-лог**: Каждое действие агента фиксируется: `agent_id`, `session_hash`, `timestamp`, `artifact_id`.

## Пример: Агент добавляет исследование
```bash
# 1. Агент читает контекст
nexus context load --pack researcher --query "volatility prediction"

# 2. Генерирует исследование
nexus create --type research_log --id research/volatility_lstm_v1 \
  --content "LSTM показывает точность 87% на исторических данных, но переобучается на шум" \
  --traces related_to:problem/openspec_monolith

# 3. Валидация (автоматическая)
nexus validate research/volatility_lstm_v1
# → Проверяет: тип в реестре, обязательные поля, формат контента

# 4. Эскалация (если критический вывод)
# → Если исследование предлагает архитектурное изменение:
nexus promote research/volatility_lstm_v1 → adr/lstm_vs_transformer_v1
# → Статус: proposed, требует человеческого аппрува

# 5. Аппрув (человек)
nexus approve adr/lstm_vs_transformer_v1
# → Статус: accepted, попадает в активный граф, доступно для атомов доставки
```
