# Журнал нюансов и эволюции концепции

> Этот файл фиксирует тонкие моменты, инсайты, изменения требований и открытые вопросы, возникающие в процессе обсуждения. Цель — отслеживать, как концепция меняется под давлением практики.

---

## 2026-04-11: Инициализация

### Нюанс #1: Атом ≠ Спека
**Инсайт**: Изначально `Atom` воспринимался как `Spec` (требование). В процессе обсуждения стало ясно: атом — это **контейнер доставки**, объединяющий спеку, ADR, тест-план, задачи и манифест.
**Влияние**: Пересмотрена структура атома, добавлены обязательные компоненты (`adr.md`, `manifest.yaml`).
**Действие**: Обновлены `principle/atom_container.md`, `requirement/atom_as_container.md`.

### Нюанс #2: Физическая стабильность > логическая чистота
**Проблема**: Физическое перемещение в архив ломает ссылки, скрывает контекст, требует ручного мерджа обобщающих документов.
**Решение**: Архив = статус, не папка. Файлы никогда не двигаются. Ссылки по `id`, резолвятся графом.
**Влияние**: Принято `ADR: logical_vs_physical_archive`.
**Действие**: Обновлены `principle/logical_archiving.md`, `requirement/id_based_navigation.md`.

### Нюанс #3: Типы не захардкожены
**Инсайт**: Жёсткая таксономия (`spark`, `note`, `atom`...) умрёт при масштабировании. Нужен конфигурируемый реестр.
**Решение**: `registry.yaml` декларирует типы, профили, FSM, шаблоны. Ядро валидирует динамически.
**Влияние**: Принято `ADR: dynamic_type_registry`.
**Действие**: Создан `meta/registry.yaml`, обновлены `principle/dynamic_registry.md`.

### Нюанс #4: Агенты — не пассивные потребители
**Проблема**: Если агенты пишут напрямую в ФС, они могут сломать целостность, обойти валидацию, создать «мусор».
**Решение**: Протокол `capture/validate/approve`. Агенты пишут через CLI/API, критические изменения требуют человеческого гейта.
**Влияние**: Принято `requirement/agent_io_protocol.md`.
**Действие**: Добавлены правила безопасности в `principle/agent_protocol.md`.

### Нюанс #5: `vault/` вместо `specs/`
**Проблема**: `specs/` слишком узко. Система хранит всё знание проекта, не только спецификации.
**Решение**: Переименовать корень в `vault/` — универсально, кратко, семантически точно.
**Влияние**: Принято `ADR: vault_over_specs`.
**Действие**: Создана структура `vault/`, удалена `specs/`.

### Нюанс #6: Обобщающие документы живые
**Проблема**: Ручной мердж в `architecture.md` при каждом изменении атомов — боль и источник ошибок.
**Решение**: Динамическая агрегация через `{{query: ...}}` шаблоны. Документ рендерится из графа, всегда актуален.
**Влияние**: Обновлен `architecture/overview.md` с примерами шаблонов.
**Действие**: Требуется реализация рендерера в ядре.

### Нюанс #7: Метафора реактора — только для людей
**Проблема**: Если использовать «реакторные» термины в CLI, это запутает агентов и усложнит парсинг.
**Решение**: В CLI/API — технические имена (`nexus atom create`). В документации, дашбордах, жаргоне — метафора (`Critical Mass`, `Chain Reaction`).
**Влияние**: Принято `ADR: reactor_metaphor`.
**Действие**: Обновлён `architecture/overview.md` с разделом «Метафора Реактора».

### Нюанс #8: Параллельность требует изоляции
**Проблема**: В одном процессе IDE нельзя запустить несколько агентов в разных ветках — конфликт `.git/index`, LSP, FS locks.
**Решение**: `git worktree` + изолированные процессы. Каждый агент в своём worktree, IDE — только для ревью.
**Влияние**: Добавлено в `principle/git_integration.md`.
**Действие**: Требуется реализация `worktree` менеджера в ядре.

### Нюанс #9: Story = Atom, но не идентичны
**Проблема**: Если атом = стори, зачем новый термин? Не избыточно ли?
**Решение**: Story = «что/зачем» (бизнес), Atom = «как/чем» (исполнение). Они дополняют друг друга через `external_refs`.
**Влияние**: Принято `ADR: story_equals_atom`.
**Действие**: Обновлены `principle/atom_container.md`, добавлено поле `external_refs` в `_meta.yaml`.

### Нюанс #10: Детерминированное ядро > вероятностный LLM
**Инсайт**: Для трейдинга/финтеха/аудита критичен детерминизм. Вероятностные ошибки LLM недопустимы в валидации контрактов.
**Решение**: Вся валидация, маршрутизация, генерация шаблонов — без LLM. LLM — опциональный интерфейс для NL-ввода.
**Влияние**: Зафиксировано в `principle/atom_container.md`, `requirement/agent_protocol.md`.
**Действие**: Ядро реализуется на Python + Pydantic v2, LLM — опциональный адаптер.

## Открытые вопросы (на 2026-04-11)

1. **Как обрабатывать циклические зависимости в графе?** 
   - Вариант: детекция + предложение разрыва через `arch_decision` или адаптер.
   - Статус: Требует реализации в `dag.py`.

2. **Как синхронизировать `external_refs` с внешними трекерами (Jira/Linear)?**
   - Вариант: Вебхук + маппинг статусов.
   - Статус: Требует проектирования адаптера.

3. **Как реализовать `{{query: ...}}` рендеринг в обзорах?**
   - Вариант: Парсер шаблонов + графовый запрос + генерация Markdown.
   - Статус: Требует реализации рендерера.

4. **Как управлять версиями атомов (`v1` → `v2`)?**
   - Вариант: `extends` + `supersedes` + семвер в `id`.
   - Статус: Требует уточнения в `registry.yaml`.

5. **Как обрабатывать «утечку» задач за пределы атома?**
   - Вариант: Валидатор: если `tasks.md` ссылается на чужой домен → предупреждение или требование нового атома.
   - Статус: Требует реализации в валидаторе.

```
После обсуждения:

    │   ├── principles/      # Принципы архитектуры
    │   ├── architecture/    # Обобщающие документы
    │   ├── governance/      # Правила, процессы
    │   └── problems/        # Выявленные проблемы
    ├── meta/
    │   ├── registry.yaml    # Реестр типов
    │   ├── system.yaml      # Конфигурация ядра
    │   ├── requirements/    # Требования к системе
    │   └── decisions/       # ADR, принятые решения
    ├── delivery/            # Атомы доставки (фичи, баги)
    └── _archive/            # Физический архив (опционально)
```

**Тренд**: От плоской структуры к иерархической, от монолита к атомарности, от ручного мерджа к динамической агрегации.

## Выводы

1. Концепция **устойчива к изменениям**: динамический реестр, логическая архивация, ID-навигация позволяют адаптироваться без ломки.
2. **Атомарность + граф + манифест** = детерминированная доставка, трассируемость, безопасная параллельность.
3. **Агенты — партнёры, не волшебники**: строгий протокол ввода/вывода, валидация, человеческий гейт для критических изменений.
4. **Метафора — инструмент, не догма**: помогает усвоить архитектуру, но не заменяет технические имена в коде.

**Следующий шаг**: Реализация ядра (парсер `vault/`, индексатор, резолвер ссылок, валидатор против `registry.yaml`).

## 2026-04-11: Анализ структуры и клиентский путь (добавлено)

### Нюанс #11: Структура логична, но требует ядра
**Инсайт**: Текущая `vault/` — отличная документация, но без парсера/валидатора/CLI это «мёртвые файлы». Ценность раскрывается только при реализации ядра.
**Влияние**: Приоритет смещается с «ещё больше документов» на «минимальное рабочее ядро».
**Действие**: Создан `meta/structure_analysis.md` с детальной оценкой. Следующий коммит — парсер `vault/`.

### Нюанс #12: `funnel/` для быстрого захвата — критичен для принятия системой
**Проблема**: Если для добавления идеи нужно заполнять `_meta.yaml`, `traces`, `id` — пользователи будут саботировать процесс.
**Решение**: `vault/context/funnel/` + шаблон с минимальным фронтматтером. CLI `nexus capture` — одна команда, всё остальное авто-генерится.
**Влияние**: Снижает порог входа, увеличивает поток идей.
**Действие**: Создана папка `funnel/` + `_template.md`.

### Нюанс #13: Связи сейчас = вручную, далее = автоматически
**Инсайт**: Пока нет ядра, связи (`[[id]]`, `traces.*`) — ручная работа. Это оверхед, но он заставляет думать о связях.
**Решение**: Ядро автоматически:
- Резолвит `[[id]]` → путь + статус + превью
- Строит граф по `traces.*`
- Валидирует битые ссылки при коммите
- Показывает обратные ссылки («кто ссылается на этот узел»)
**Влияние**: Ручная работа сейчас → инвестиция в качество графа. Автоматизация потом → масштабирование без потери целостности.
**Действие**: Зафиксировано в `meta/structure_analysis.md`.

### Нюанс #14: Клиентский путь должен быть многоскоростным
**Инсайт**: Разные сценарии требуют разного уровня детализации:
- Быстрая идея: 30 секунд, `funnel/`, `status: raw`
- Исследование: несколько дней, `research/`, `status: analyzed`
- Атом доставки: спринт, `delivery/`, `status: approved → completed`
**Решение**: Не форсировать все артефакты через строгий `delivery`-профиль. Динамический реестр позволяет гибкие профили.
**Влияние**: Система не душит креативность бюрократией, но гарантирует качество на этапе доставки.
**Действие**: Описано в `meta/structure_analysis.md`, реализовано в `registry.yaml` (профили `delivery`/`context`).
