# Динамический реестр типов

## Тезис
Типы артефактов не захардкожены. Они конфигурируются через `registry/types.yaml`, что позволяет адаптировать систему под любую процессную модель без изменения ядра.

## Принцип
1. Профили валидации: `delivery` (строгий FSM + manifest), `context` (гибкий FSM + контент), `research` (экспериментальный).
2. Методологическая агностичность: `story`, `task`, `change_request`, `hypothesis` — всё это типы из реестра. Система валидирует переходы, а не навязывает термины.
3. Расширяемость: новые типы добавляются декларативно. Ядро читает схему, строит валидатор, CLI генерирует шаблоны.

## Влияние
Смена методологии = обновление `registry.yaml` + маппинг статусов. Код, атомы и граф не ломаются.
