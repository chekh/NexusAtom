# Разделение Vault (локальный) и Store (удалённый)

## Контекст
Изначально `asset_marketplace` находился внутри `vault/nuclear/`. Это создавало путаницу: локальный проект содержал удалённый каталог активов, что нарушало границы ответственности и усложняло синхронизацию.

## Решение
Чёткое разделение на два независимых ресурса:
- **`vault/`** — всегда локальный. Содержит знания, спеки, контракты и рабочую память конкретного проекта. Создаётся в корне проекта.
- **`store/`** — всегда удалённый. Реестр/маркетплейс готовых скиллов, шаблонов, профилей. В разработке моделируется папкой в корне репо. В продакшене — отдельный Git-репо, Registry Server или CDN.

## Поток данных
```
[Store (Remote)] 
    │ (nexus pull / registry API)
    ▼
[~/.nexus/cache/] (Локальный кэш пакетов)
    │ (линк/копирование)
    ▼
[vault/_core/config/skills/] (Локальная активация в проекте)
```

## Правила
1. `vault/` никогда не пушится в Store. Это проектное состояние.
2. `store/` никогда не модифируется агентами проекта. Это публичный/внутренний каталог.
3. Установка актива: `nexus install skill notebook@1.0.0` → скачивает из Store → кладёт в кэш → активирует в Vault.
4. Версионирование: Store использует строгий SemVer. Vault фиксирует версии в `lock.yaml`.

## Итог
Vault = Контекст проекта. Store = Библиотека компонентов. Разделение гарантирует чистоту, безопасность и масштабируемость экосистемы.
