# Рецензия экспертной панели: Двухуровневая модель ресурсов (System vs Nuclear)

## Состав панели (5 экспертов, нечётное число для избежания ничьих)

| Эксперт | Роль | Фокус критики |
|---------|------|---------------|
| **Алиса (Системный архитектор)** | Масштабируемость, контракты, миграция | Изоляция системного/ядерного, версионирование схемы, производительность |
| **Боб (Специалист по управлению знаниями)** | Юзабилити, агентская память, семантика | Структура памяти, навигация, когнитивная нагрузка, эволюция онтологии |

## 🔍 Критика Алисы (Системный архитектор)

### ✅ Сильные стороны
- **Чёткое разделение ответственности**: Системные ресурсы (`_system/`, `context/`, `meta/`) не смешиваются с доменными (`nuclear/`). Упрощает поддержку и эволюцию.
- **Контрактная модель ядерных ресурсов**: Явное определение в `schema.yaml` позволяет инструментам валидировать соответствие до записи.
- **Адаптивность через конфигурацию**: Добавление нового домена = создание папки + запись в `schema.yaml`, не изменение кода.

### ⚠️ Риски и вопросы
1.  **Миграция существующих `vault/`**: Как перенести текущую структуру (`context/`, `meta/`, `delivery/`) в новую модель без потери трассируемости?
    - *Предложение*: Создать скрипт `vault migrate to-system-nuclear`, который:
      - Создаёт `_system/manifest.yaml` и `schema.yaml` на основе текущего `meta/registry.yaml`.
      - Перемещает `delivery/` → `nuclear/engineering/`, `context/principles/` → `context/principles/` (остаётся), `funnel/` → `workspace/inbox/`.
      - Обновляет `id` артефактов, если они включали старые пути (опционально, с флагом `--preserve-ids`).
2.  **Версионирование контрактов ядерных ресурсов**: Что если `schema.yaml` меняется, а ядерный ресурс использует старую версию типа?
    - *Предложение*: В `schema.yaml` поддерживать `supported_type_versions`. Ядро должно уметь работать с несколькими версиями типов параллельно, предоставляя миграционные скрипты для артефактов.
3.  **Производительность индексации при множестве ядерных ресурсов**: При 50+ доменах сканирование `nuclear/` для построения индекса может занять время.
    - *Предложение*: Инкрементальная индексация по событиям (git hook, file watcher). Кэширование метаданных в SQLite с шардированием по ядерным ресурсам.

### 🎯 Рекомендации
- Добавить в `_system/manifest.yaml` поле `migration_scripts`, указывающее на скрипты для перехода между версиями структуры.
- Реализовать «ленивую» индексацию ядерных ресурсов: индекс строится по мере доступа, не сканируя всё при старте.
- Для больших `vault/` поддержать шардирование индекса по `nuclear/` доменам.

## 🔍 Критика Боба (Специалист по управлению знаниями)

### ✅ Сильные стороны
- **Агентская память как первый класс артефакт**: Структура `context/agents/{id}/` делает память версионируемой, экспортируемой, трассируемой.
- **Гибкость `nuclear/`**: Позволяет команде создать пространство, которое отражает их ментальную модель, а не навязанную структуру.
- **`context/templates/` для онбординга**: Шаблоны артефактов по типам упрощают начало работы.

### ⚠️ Риски и вопросы
1.  **Когнитивная перегрузка от структуры памяти агентов**: Если память делится на `short_term/`, `long_term/`, `session_summaries/`, агент (или человек) может запутаться, куда что писать.
    - *Предложение*: Создать `context/templates/agent_memory_guide.md` с примерами: «Куда сохранять вывод сессии?», «Когда компрессировать в summary?». CLI `agent save-memory --type summary` для упрощения.
2.  **Эволюция онтологии ядерных ресурсов**: Как добавлять новые типы в `schema.yaml`, не ломая существующие артефакты в `nuclear/`?
    - *Предложение*: Использовать семвер для типов. Новый тип `research_log.v2` может сосуществовать с `research_log.v1`. Устаревшие типы помечаются `deprecated` в схеме, но не удаляются.
3.  **Поиск по всему `vault/` vs по ядерному ресурсу**: Как пользователю понять, где искать? По умолчанию искать везде или только в активном `nuclear/`?
    - *Предложение*: CLI `vault search` по умолчанию ищет во всех `nuclear/`, но поддерживает фильтр `--nuclear engineering`. UI должен явно показывать, в каком пространстве выполняется поиск.

### 🎯 Рекомендации
- Создать `context/templates/nuclear_starter_kit.md` с примерами конфигурации ядерных ресурсов для типовых доменов (софт, исследование, бизнес).
- В CLI добавить команду `vault nuclear list --with-stats`, показывающую количество артефактов, типов, активность в каждом ресурсе.
- Реализовать «умный поиск», который ранжирует результаты из активного `nuclear/` выше, но не скрывает другие.

## 🔍 Критика Кэрол (Инженер по безопасности и DevOps)

### ✅ Сильные стороны
- **Явная ACL в `schema.yaml`**: Права доступа определены декларативно, не в коде. Легко аудитировать и менять.
- **Изоляция `_system/`**: Системные конфиги не могут быть изменены агентами с ролью `builder` (только `architect`).
- **Контракт на ядерные ресурсы**: Валидация соответствия при коммите предотвращает «дрейф» структуры.

### ⚠️ Риски и вопросы
1.  **Безопасность агентской памяти**: Память может содержать чувствительные данные (ключи, персональные данные). Как защитить?
    - *Предложение*: Добавить в `acl.yaml` правило `encrypt_agent_memory: true` для шифрования `context/agents/` на уровне ФС. Опция `clear_memory_on_session_end` для автоматической очистки краткосрочной памяти.
2.  **Аудит изменений ядерных ресурсов**: Кто и когда изменил контракт ядерного ресурса в `schema.yaml`? Как отследить?
    - *Предложение*: Все изменения в `_system/schema.yaml` должны коммититься с обязательным `commit message` формата `chore(schema): update nuclear resource X`. Интеграция с Git history + отдельный `meta/audit/schema_changes.log`.
3.  **Целостность при параллельной работе**: Если два агента одновременно меняют `schema.yaml`, как разрешается конфликт?
    - *Предложение*: Блокировка на запись `_system/` через optimistic locking. При конфликте — требование ручного мерджа с аппрувом `architect`.

### 🎯 Рекомендации
- Добавить в `_system/` файл `audit_policy.yaml`, определяющий, какие события логировать (изменение схемы, доступ к `context/agents/`, импорт/экспорт ядерных ресурсов).
- Реализовать `vault audit --last-24h --nuclear engineering` для быстрого просмотра активности.
- Для критических ядерных ресурсов (например, `nuclear/security/`) поддержать требование `multi-signature approval` для изменений контракта (2 из 3 `architect` должны аппрувить).

## 🔍 Критика Дэвида (AI/LLM Engineer)

### ✅ Сильные стороны
- **Структурированная память агентов**: Разделение на `short_term/`, `long_term/`, `session_summaries/` идеально для управления контекстным окном.
- **Интеграция с RAG**: Векторизация `context/agents/` и `nuclear/` позволяет агенту быстро находить релевантные знания.
- **Контрактная гарантия**: Агент не может записать артефакт, не соответствующий `schema.yaml` — предотвращает «галлюцинации» в структуре.

### ⚠️ Риски и вопросы
1.  **Векторизация памяти**: Как определять, какие части памяти векторизовать для семантического поиска? Вся `long_term/`? Только `session_summaries/`?
    - *Предложение*: Добавить в `schema.yaml` поле `vectorization_policy` для каждого типа памяти: `vectorize: [long_term, session_summaries]`, `exclude_fields: [raw_logs]`.
2.  **Управление контекстным окном**: Если агент загружает память + ядерный ресурс, контекст может переполниться.
    - *Предложение*: Реализовать «контекстный бюджет» в `schema.yaml`: `max_context_tokens: 8000`. Ядро автоматически обрезает наименее релевантные части памяти при загрузке.
3.  **Семантический поиск по ядерным ресурсам**: Как агенту найти «все атомы, связанные с аутентификацией», если они разбросаны по `nuclear/engineering/`, `nuclear/security/`?
    - *Предложение*: Поддержать кросс-ядерный семантический индекс. Запрос `vault search --semantic "аутентификация"` возвращает результаты из всех `nuclear/`, ранжированные по релевантности.

### 🎯 Рекомендации
- Добавить в `schema.yaml` конфигурацию для RAG: `rag_config: {embedding_model: "text-embedding-3-small", chunk_size: 512, overlap: 50}`.
- Реализовать `agent context-budget` команду для мониторинга использования токенов.
- Поддержать «умную подгрузку» памяти: агент загружает только релевантные части `long_term/` на основе текущего запроса.

## 🔍 Критика Елены (Product/Process Strategist)

### ✅ Сильные стороны
- **Спектр адаптивности**: Модель покрывает всё — от записной книжки до энтерпрайза. Не требует «перепроектирования» при росте проекта.
- **Низкий порог входа для малых проектов**: Личная записная книжка = 1 ядерный ресурс, минимум конфигов.
- **Эволюция без разрывов**: Изменение `schema.yaml` не ломает старые артефакты.

### ⚠️ Риски и вопросы
1.  **Риск over-engineering для малых проектов**: Если по умолчанию `vault/` включает 5 системных ресурсов + сложную `schema.yaml`, пользователь может испугаться.
    - *Предложение*: Создать «упрощённый профиль» в `manifest.yaml`: `profile: minimal`. При этом профиле `schema.yaml` содержит только базовые типы (`note`, `inquiry`), а ядерные ресурсы не требуют явного контракта.
2.  **Онбординг новых участников**: Как объяснить двухуровневую модель человеку, который видит `vault/` впервые?
    - *Предложение*: Создать интерактивный `context/templates/vault_tour.md` с примерами: «Вот как выглядит записная книжка», «Вот как выглядит dev-проект». CLI `vault tour --profile notebook` для демонстрации.
3.  **Эволюция методологии**: Как команда понимает, когда нужно добавить новый ядерный ресурс или изменить `schema.yaml`?
    - *Предложение*: Ввести «ретроспективу структуры» раз в квартал: команда оценивает, хватает ли текущих ядерных ресурсов, не стала ли `schema.yaml` слишком сложной.

### 🎯 Рекомендации
- Добавить в `_system/manifest.yaml` поле `profiles: [minimal, standard, enterprise]` с предустановленными конфигурациями `schema.yaml`.
- Создать `context/templates/vault_migration_path.md`, показывающий, как эволюционировать от `profile: minimal` к `profile: enterprise` пошагово.
- Реализовать `vault health-check` команду, которая оценивает сложность `vault/` и предлагает упрощения или расширения.
