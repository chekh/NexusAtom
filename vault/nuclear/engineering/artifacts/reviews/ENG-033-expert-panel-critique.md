# Рецензия экспертной панели: Структура `vault/` как независимый ресурс

## Состав панели

| Эксперт | Роль | Фокус критики |
|---------|------|---------------|
| **Алиса (Системный архитектор)** | Проектирование масштабируемых, портативных систем | Изоляция, версионирование, миграция, производительность |
| **Боб (Специалист по управлению знаниями)** | Юзабилити, поиск, когнитивная нагрузка, онтологии | Навигация, классификация, ментальные модели, адаптивность |
| **Кэрол (Инженер по безопасности и DevOps)** | Контроль доступа, аудит, целостность, CI/CD интеграция | ACL, логирование, верификация, автоматизация |

## 🔍 Критика Алисы (Системный архитектор)

### ✅ Сильные стороны
- **Само-описываемость через `manifest.yaml`**: Чёткий контракт на ресурс. Позволяет инструментам автоматически парсить и валидировать `vault/`.
- **Разделение `_system/` и `namespaces/`**: Изоляция системных конфигов от доменных данных. Упрощает обновление схемы без риска сломать контент.
- **Гибкость `namespaces/`**: Позволяет адаптировать структуру под любой домен без изменения ядра.

### ⚠️ Риски и вопросы
1.  **Версионирование схемы**: Как обрабатывать артефакты, созданные по `schema.yaml v1`, когда ресурс обновлён до `v2`?
    - *Предложение*: В `manifest.yaml` добавить `supported_schema_versions: [1.0, 2.0]`. Ядро должно уметь работать с несколькими версиями схемы параллельно, предоставляя миграционные скрипты.
2.  **Производительность индексации**: При 10000+ артефактов сканирование `namespaces/` для построения индекса может занять время.
    - *Предложение*: Инкрементальная индексация по событиям (git hook, file watcher). Кэширование метаданных в SQLite с TTL.
3.  **Миграция с v1 на v2**: Предложенный план требует ручного перемещения файлов. Для больших `vault/` это рискованно.
    - *Предложение*: Создать скрипт `vault migrate v1→v2`, который автоматически переименовывает пути, обновляет `id` в метаданных и создаёт бэкап.

### 🎯 Рекомендации
- Добавить в `manifest.yaml` поле `migration_scripts`, указывающее на скрипты для перехода между версиями схемы.
- Реализовать «ленивую» индексацию: индекс строится по мере доступа к артефактам, не сканируя всё при старте.
- Для больших ресурсов поддержать шардирование индекса по `namespaces/`.

## 🔍 Критика Боба (Специалист по управлению знаниями)

### ✅ Сильные стороны
- **`workspace/inbox/` для быстрого захвата**: Низкий порог входа. Не нужно сразу определять `namespace` или `type`.
- **`shared/` для кросс-доменных знаний**: Глоссарий, шаблоны, онтологии в одном месте. Упрощает согласованность терминологии.
- **Гибкие `namespaces/`**: Позволяет команде создать пространство, которое отражает их ментальную модель, а не навязанную структуру.

### ⚠️ Риски и вопросы
1.  **Когнитивная перегрузка от гибкости**: Если каждый проект создаёт свои `namespaces/`, новым участникам сложно ориентироваться.
    - *Предложение*: В `schema.yaml` определить `recommended_namespaces` для типовых сценариев (софт, исследование, бизнес). Разрешить кастомные, но с предупреждением.
2.  **Поиск по всему `vault/` vs по `namespace`**: Как пользователю понять, где искать? По умолчанию искать везде или только в активном пространстве?
    - *Предложение*: CLI `vault search` по умолчанию ищет во всех `namespaces/`, но поддерживает фильтр `--namespace engineering`. UI должен явно показывать, в каком пространстве выполняется поиск.
3.  **Эволюция онтологии**: Как добавлять новые типы в `schema.yaml`, не ломая существующие артефакты?
    - *Предложение*: Использовать семвер для типов. Новый тип `research_log.v2` может сосуществовать с `research_log.v1`. Устаревшие типы помечаются `deprecated` в схеме, но не удаляются.

### 🎯 Рекомендации
- Создать `shared/templates/namespace_starter_kit.md` с примерами конфигурации `namespaces/` для разных доменов.
- В CLI добавить команду `vault namespaces list --with-stats`, показывающую количество артефактов, типов, активность в каждом пространстве.
- Реализовать «умный поиск», который ранжирует результаты из активного `namespace` выше, но не скрывает другие.

## 🔍 Критика Кэрол (Инженер по безопасности и DevOps)

### ✅ Сильные стороны
- **Явная ACL в `schema.yaml`**: Права доступа определены декларативно, не в коде. Легко аудитировать и менять.
- **Изоляция `_system/`**: Системные конфиги не могут быть изменены агентами с ролью `builder` (только `architect`).
- **Контракт на импорт/экспорт**: Чёткие команды `vault import/export` с валидацией. Не позволяет случайно засорить `vault/` невалидными данными.

### ⚠️ Риски и вопросы
1.  **Целостность при параллельной работе**: Если два агента одновременно меняют `schema.yaml`, как разрешается конфликт?
    - *Предложение*: Блокировка на запись `_system/` через optimistic locking. При конфликте — требование ручного мерджа с аппрувом `architect`.
2.  **Аудит изменений схемы**: Кто и когда изменил `acl.yaml` или добавил новый тип? Как отследить?
    - *Предложение*: Все изменения в `_system/` должны коммититься с обязательным `commit message` формата `chore(schema): add type X`. Интеграция с Git history + отдельный `audit_log.md` в `_system/`.
3.  **Безопасность импорта**: Что если импортируемый файл содержит вредоносный контент или битые ссылки?
    - *Предложение*: `vault import` запускает валидацию против `schema.yaml` перед записью. Битые ссылки → ошибка. Вредоносный контент (скрипты) → карантин в `workspace/quarantine/` с алертом.

### 🎯 Рекомендации
- Добавить в `_system/` файл `audit_policy.yaml`, определяющий, какие события логировать (изменение схемы, доступ к чувствительным `namespaces/`, импорт/экспорт).
- Реализовать `vault audit --last-24h --namespace engineering` для быстрого просмотра активности.
- Для критических `namespaces/` (например, `security/`) поддержать требование `multi-signature approval` для изменений (2 из 3 `architect` должны аппрувить).

## 🎯 Сводные рекомендации панели

### 🔴 Критические (блокируют принятие спецификации)
1.  **Механизм миграции схемы**: Добавить в `manifest.yaml` поддержку `supported_schema_versions` и скриптов миграции.
2.  **Аудит изменений `_system/`**: Реализовать логирование и обязательный формат коммитов для системных файлов.
3.  **Валидация при импорте**: `vault import` должен отклонять невалидные артефакты до записи в `vault/`.

### 🟡 Важные (требуют доработки перед реализацией)
4.  **Умный поиск с контекстом `namespace`**: CLI/UI должны явно показывать, в каком пространстве выполняется поиск, и ранжировать результаты.
5.  **Шаблон для `namespaces/`**: Создать стартовый набор конфигураций для типовых доменов, чтобы снизить когнитивную нагрузку.
6.  **Оптимистичная блокировка для `_system/`**: Предотвращение конфликтов при параллельном редактировании схемы.

### 🟢 Желательные (улучшения после базовой реализации)
7.  **Шардирование индекса по `namespaces/`** для больших ресурсов.
8.  **Мульти-сигнатурный аппрув** для критических пространств.
9.  **Визуализация графа зависимостей** между `namespaces/`.
