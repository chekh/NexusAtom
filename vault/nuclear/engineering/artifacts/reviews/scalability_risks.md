# Риски масштабируемости: Что сломается при 1000+ атомах?

## Сценарии масштабирования

| Сценарий | Текущее состояние | Риск при 1000 атомов | Риск при 10000 атомов |
|----------|------------------|---------------------|----------------------|
| **Поиск по `id`** | Ручной поиск по файлам или `grep` | 🟡 Медленно, но терпимо | 🔴 Неприемлемо: линейный поиск по 10000 файлов |
| **Резолвинг `[[id]]`** | Текстовые ссылки, не функциональные | 🟡 Можно парсить на лету | 🔴 Требует индекса, иначе рендеринг минут/часов |
| **Валидация `registry.yaml`** | Ручная проверка человеком | 🟡 Ошибки вероятны, но локализуемы | 🔴 Человеческая валидация не масштабируется |
| **Построение графа зависимостей** | Вручную через `traces.*` | 🟡 Граф можно нарисовать на бумаге | 🔴 Требуется графовая БД или in-memory индекс |
| **Архивация (логическая)** | `status: archived` в YAML | 🟡 Фильтрация по статусу в коде | 🔴 Индексатор должен кэшировать статусы, иначе каждый запрос сканирует всё |
| **Параллельная работа агентов** | Нет механизма блокировок | 🟡 Редкие конфликты, ручное разрешение | 🔴 Частые коллизии, требуется оптимистичная блокировка + merge-стратегия |
| **Хранение `manifest.yaml` с хэшами** | Текстовый файл в атоме | 🟡 Можно проверять при коммите | 🔴 Проверка всех манифестов при каждом коммите = минуты/часы |

## Критические точки отказа

### 1. Отсутствие индекса
> `principle/vault_ssot`: «Всё — узлы единого графа.»

❌ **Проблема**: Граф существует только концептуально. Нет индекса → каждый поиск = сканирование всех файлов.

📈 **Прогноз**:
- 100 атомов: ~1-2 секунды на поиск (терпимо)
- 1000 атомов: ~10-20 секунд (раздражает)
- 10000 атомов: ~2-5 минут (неприемлемо)

✅ **Решение**: In-memory индекс (SQLite / Redis) с автообновлением при коммитах.

### 2. Линейная валидация
> `principle/dynamic_registry`: «Система валидирует переходы...»

❌ **Проблема**: Валидация каждого артефакта требует загрузки `registry.yaml` и проверки FSM. При 10000 атомах и частых изменениях — нагрузка растёт линейно.

📈 **Прогноз**:
- Ручная валидация: не масштабируется вообще
- Автоматическая, но последовательная: ~100 атомов/минуту → 10000 = 100 минут
- Автоматическая + параллельная: требует координации, риск гонок

✅ **Решение**: 
- Кэширование схемы `registry.yaml`
- Параллельная валидация с оптимистичными блокировками
- Инкрементальная валидация (только изменённые узлы)

### 3. Граф зависимостей в памяти
> `structure_analysis.md`: «Связность: высокая. Каждый файл имеет `traces.*`... Граф строится автоматически.»

❌ **Проблема**: «Автоматически» = кем? Если это делает агент/человек вручную — не масштабируется. Если ядро — где код?

📈 **Прогноз**:
- Ручное поддержание графа: максимум 50-100 узлов
- Автоматическое, но без индекса: 500-1000 узлов (память + время)
- Автоматическое + индекс + графовая БД: 10000+ узлов (приемлемо)

✅ **Решение**: 
- Графовый движок (NetworkX для начала, Neo4j для продакшена)
- Ленивая загрузка подграфов (не весь граф в памяти)
- Кэширование часто запрашиваемых путей

### 4. Конфликты при параллельной работе
> `governance/git_spec_first_trunk.md`: «Агенты пишут только в свои feat/ ветки. Конфликты решаются через rebase --onto spec/{id}.»

❌ **Проблема**: `rebase --onto` — это решение на уровне Git, не на уровне методологии. Если два агента меняют один и тот же `manifest.yaml` — кто прав?

📈 **Прогноз**:
- 1-2 агента: редкие конфликты, ручное разрешение
- 10+ агентов: частые конфликты, требуется автоматическое разрешение
- 100+ агентов: без стратегии разрешения конфликтов — хаос

✅ **Решение**:
- Оптимистичные блокировки на уровне атома
- Автоматический мердж манифестов по `content_hash`
- Человеческий арбитраж только при неавтоматически разрешимых коллизиях

### 5. Логическая архивация без индекса
> `principle/logical_archiving`: «Файлы никогда не перемещаются. Меняется только `status`...»

❌ **Проблема**: Если поиск по умолчанию исключает `archived`, но индексатор сканирует все файлы — производительность падает с ростом архива.

📈 **Прогноз**:
- 10% архива: незначительное влияние
- 50% архива: сканирование 2× больше файлов
- 90% архива: 90% времени тратится на фильтрацию, не на поиск

✅ **Решение**:
- Индекс хранит `status` как отдельное поле
- Поиск по умолчанию использует индекс, не файловую систему
- Физическая архивация (`nexus vault cleanup --physical`) как опция для энтерпрайза

## Матрица рисков

| Риск | Вероятность | Влияние | Приоритет | Митигация |
|------|-------------|---------|-----------|-----------|
| Отсутствие индекса | 🔴 Высокая | 🔴 Критическое | 🔴 Высокий | Создать индексатор в Фазе 0 ядра |
| Линейная валидация | 🟡 Средняя | 🔴 Критическое | 🔴 Высокий | Параллельная валидация + кэш схемы |
| Граф в памяти | 🟡 Средняя | 🟡 Серьёзное | 🟡 Средний | Ленивая загрузка + NetworkX |
| Конфликты агентов | 🟡 Средняя | 🟡 Серьёзное | 🟡 Средний | Оптимистичные блокировки |
| Архивация без индекса | 🟢 Низкая (пока) | 🟡 Серьёзное | 🟢 Низкий | Отложить до 1000+ атомов |

## Рекомендации по приоритетам

### 🔴 До 100 атомов (сейчас)
1. Создать минимальный индексатор (SQLite + in-memory кэш)
2. Реализовать параллельную валидацию с кэшированием `registry.yaml`
3. Добавить `nexus validate --incremental` для проверки только изменённых узлов

### 🟡 До 1000 атомов
4. Интегрировать NetworkX для построения и запросов к графу
5. Реализовать оптимистичные блокировки на уровне атома
6. Добавить мониторинг производительности (метрики: время поиска, валидации, построения графа)

### 🟢 До 10000 атомов
7. Миграция на графовую БД (Neo4j / Amazon Neptune) при необходимости
8. Горизонтальное масштабирование индексатора (шардирование по доменам)
9. Автоматическая физическая архивация старых узлов с сохранением метаданных

## Вердикт
Методология **теоретически масштабируема**, но **практически не готова к масштабированию** без реализации ядра. 
Критические риски (индекс, валидация, граф) требуют решения до достижения 100 атомов.
