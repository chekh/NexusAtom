## 🔍 Детальный разбор находок агента

### 1. Критический пробел: отсутствие ядра
**Артефакт**: `ENG-031`

| Проблема | Влияние | Решение |
|----------|---------|---------|
| Нет парсера `vault/` | Агенты не могут безопасно читать метаданные | Индексатор: сканирование → SQLite-индекс |
| Нет валидатора | Противоречия в `registry.yaml` не детектируются | Валидатор: проверка FSM, обязательных полей |
| Нет CLI | Ручная работа с файлами = ошибки, оверхед | CLI: `query`, `resolve`, `validate`, `context load` |

**Статус**: 🔴 **Блокирующий**. Без ядра методология остаётся «бумажной».

### 2. Противоречие: «Агностичность» vs «Жёсткий FSM»
**Артефакт**: `ENG-027`

| Источник А | Источник Б | Конфликт |
|-----------|-----------|----------|
| `principle/dynamic_registry`: «Система валидирует переходы, а не навязывает термины» | `principle/agent_protocol`: «Агенты не обходят гейты. Критические изменения проходят Review Gate» | Если система агностична, почему агенты ограничены? Где граница гибкости? |

**Решение**: Добавить в `registry.yaml` явное поле `agent_permissions`:
```yaml
types:
  atom:
    profile: delivery
    fsm: [draft, rfc, approved, ...]
    fsm_strict: true
    agent_permissions:
      can_create: [draft]
      can_transition: [draft→rfc]  # только вперёд, без пропуска
      requires_human_approval: [rfc→approved, approved→in_progress]
```

**Статус**: 🟡 **Требует уточнения в реестре**. Не блокирует ядро, но критичен для агентов.

### 3. Риск масштабируемости: перегрузка контекста
**Артефакт**: `ENG-038`

| Сценарий | Без защиты | С протоколом изоляции |
|----------|------------|----------------------|
| Поиск при 10000 атомах | ~2-5 минут (линейный скан) | ~50 мс (индекс) |
| Контекст агента | ~100-500 МБ (все файлы) | ~30-50 КБ (метаданные + 1 атом) |
| Валидация при коммите | Минуты (последовательно) | Секунды (параллельно + кэш) |

**Решение**: Принят `GOV.PROTO.CONTEXT_ISOLATION.v1` (см. ниже).

**Статус**: 🟢 **Решено архитектурно**. Требует реализации в ядре.

## ✅ Принятые решения

### 1. Протокол изоляции контекста
**Артефакт**: `GOV.PROTO.CONTEXT_ISOLATION.v1`

**Суть**: Агент не читает `vault/` напрямую. Все операции — через ядро с фильтрацией.

| Механизм | Реализация | Приоритет |
|----------|------------|-----------|
| Индекс-первый | SQLite-индекс метаданных (не контента) | 🔴 Критический |
| Ленивая загрузка | Контент по `id` только при `resolve` | 🟡 Высокий |
| Контекст-паки | Предопределённые наборы под роль | 🟡 Высокий |
| ACL по ролям | `read_roles` в `registry.yaml` | 🟡 Высокий |
| Запрос-ориентированный доступ | `nexus query --filter`, не `grep` | 🔴 Критический |

**Влияние**: Защищает от перегрузки контекста, обеспечивает безопасность, масштабируемость.

### 2. Уточнение реестра типов
**Действие**: Дополнить `meta/registry.yaml` полями:
```yaml
profiles:
  delivery:
    agent_permissions:
      can_create: [draft]
      can_transition: [draft→rfc]
      requires_human_approval: [rfc→approved]
  context:
    agent_permissions:
      can_create: [raw]
      can_transition: [raw→refined]
      requires_human_approval: []  # контекст можно менять свободно
```

**Статус**: 🟡 В работе (после реализации ядра).

### 3. Приоритизация: Фаза 0 (Минимальное ядро)
**Решение**: Приостановить развитие документации до создания ядра.

**Состав Фазы 0**:
```
core/
├── indexer/      # Сканирование vault/ → SQLite (metadata only)
├── validator/    # Проверка против registry.yaml (FSM, поля, связи)
├── resolver/     # Ленивая загрузка контента, резолвинг [[id]]
├── router/       # Применение ACL, фильтрация по ролям
└── cli/          # nexus query, resolve, validate, context load
```

**Критерии завершения Фазы 0**:
- [ ] Индексатор строит индекс за <1с при 100 атомах
- [ ] Валидатор отклоняет артефакт с нарушенным FSM
- [ ] CLI возвращает только метаданные при `query`, контент при `resolve`
- [ ] Попытка прочитать запрещённый тип → `AccessDenied`

## 🎯 Критерии успеха Фазы 0

1.  **Безопасность**: Агент не может прочитать запрещённый тип → `AccessDenied`.
2.  **Производительность**: Поиск по 100 атомам <1с, по 1000 атомам <5с.
3.  **Целостность**: Валидатор отклоняет артефакт с битой ссылкой или нарушенным FSM.
4.  **Удобство**: `nexus query --filter "type=atom AND status=approved"` возвращает релевантные метаданные.
5.  **Масштабируемость**: Индекс обновляется инкрементально при коммитах, не пересчитывается целиком.
