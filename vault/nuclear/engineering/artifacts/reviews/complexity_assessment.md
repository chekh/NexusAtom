# Оценка сложности методологии NexusAtom

## Executive Summary
Методология **сложна для онбординга нового участника за 30 минут**. Требуется ~2-3 часа для базового понимания и ~1 день для продуктивной работы. Для агентов сложность выше из-за необходимости интерпретировать многоуровневые правила.

## Оценка по аспектам (шкала 1-10, где 10 = максимально сложно)

| Аспект | Оценка | Комментарий |
|--------|--------|-------------|
| **Онбординг человека** | 8 | Требуется изучить: 6 принципов, 4 ADR, registry.yaml, governance-правила, структуру `vault/`. Без CLI-инструментов — ручной поиск и понимание фронтматтера. |
| **Онбординг агента** | 9 | Агент должен парсить YAML-фронтматтер, интерпретировать `traces.*`, соблюдать FSM-переходы, понимать контекст-паки. Ошибки в интерпретации → поломка графа. |
| **Повседневная работа (человек)** | 6 | После онбординга: создание атомов по шаблону, использование `[[id]]`-ссылок, навигация по статусам. Но отсутствие CLI усложняет рутину. |
| **Повседневная работа (агент)** | 7 | Агент может генерировать артефакты, но требует чётких границ: что можно менять, какие статусы доступны, когда эскалировать. |
| **Отладка и аудит** | 5 | Логическая архивация и трассировка через `traces.*` упрощают аудит. Но отсутствие инструмента визуализации графа затрудняет навигацию. |
| **Масштабирование процесса** | 7 | Динамический реестр позволяет добавлять типы, но каждый новый тип требует обновления документации, шаблонов и, возможно, CLI. |

## Детальный разбор источников сложности

### 1. Многоуровневая структура `vault/`
```
vault/
├── context/    # Знание, правила, проблемы
├── meta/       # Управление системой
└── delivery/   # Исполнение
```
✅ Логичное разделение.
❌ Новому участнику нужно понять, куда класть каждый тип артефакта. Таблица в `structure_analysis.md` помогает, но требует запоминания.

### 2. Фронтматтер + `_meta.yaml`
- Некоторые файлы используют YAML-фронтматтер в Markdown.
- Для `delivery`-атомов требуется отдельный `_meta.yaml`.
❓ **Вопрос**: Почему два формата? Не создаёт ли это когнитивную нагрузку?

### 3. FSM с 8 состояниями для атомов
```
[draft, rfc, approved, in_progress, completed, partial, deferred, archived]
```
✅ Гибкость для разных сценариев.
❌ Новичок должен запомнить допустимые переходы. Без визуальной диаграммы — легко ошибиться.

### 4. Система ссылок `[[id]]` + `traces.*`
✅ Мощная трассировка, не зависит от физического пути.
❌ Требует понимания разницы между:
   - `traces.related_to` (связь)
   - `traces.resolves` (решение проблемы)
   - `traces.supersedes` (замена)
   - `[[id]]` в контенте (ссылка для чтения)

### 5. Профили валидации (`delivery` vs `context`)
```yaml
profiles:
  delivery:
    fsm_strict: true
    ci_gate: required
  context:
    fsm_strict: false
    ci_gate: optional
```
✅ Разная строгость для разных типов артефактов.
❌ Агент должен «знать», к какому профилю относится тип, чтобы применять правильные правила.

### 6. Агентский протокол с 4 уровнями ограничений
| Уровень | Что можно | Что нельзя |
|---------|-----------|------------|
| Чтение | Все узлы с `status: active/approved` | `raw`/`draft` без флага |
| Захват | Создание `raw`-заметок | Прямая запись в `vault/` |
| Генерация | Черновики атомов | Изменение утверждённых контрактов |
| Эскалация | Предложение изменений | Самостоятельный аппрув критических типов |

❌ Много правил, легко нарушить случайно.

## Рекомендации по снижению сложности

### 🔴 Высокий приоритет
1. **Создать визуальную диаграмму FSM** для каждого типа в `registry.yaml` (можно генерировать автоматически).
2. **Добавить CLI-команду `nexus guide --role <role>`**, которая выводит чек-лист действий для роли (новый участник, агент, аудитор).
3. **Унифицировать метаданные**: либо всегда фронтматтер в `.md`, либо всегда `_meta.yaml`. Два формата увеличивают когнитивную нагрузку.

### 🟡 Средний приоритет
4. **Создать интерактивный туториал** в `vault/context/onboarding/` с пошаговыми сценариями.
5. **Добавить команду `nexus validate --dry-run`**, которая показывает, какие проверки пройдут/не пройдут, без реального изменения статуса.
6. **Генерировать «шпаргалку»** по `traces.*` полям с примерами для каждого типа связи.

### 🟢 Низкий приоритет
7. **Добавить графический дашборд** для визуализации связей между атомами (позже, когда будет ядро).
8. **Создать «песочницу»** — изолированную копию `vault/` для обучения без риска поломки основного графа.

## Вердикт
Методология **мощная, но перегружена деталями** для быстрого старта. Рекомендуется:
1. Создать минимальный рабочий набор правил (MVP-онбординг).
2. Постепенно раскрывать сложность по мере роста компетенции участника.
3. Инвестировать в CLI и визуализацию — они снизят порог входа на 30-40%.
