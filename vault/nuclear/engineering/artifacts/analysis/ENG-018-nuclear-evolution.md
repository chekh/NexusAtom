# Анализ: Эволюция ядерных ресурсов (Локальные контракты, Память, Зоны тьмы)

## Тезисы для обсуждения

### 1. Локальные контракты ядерных ресурсов
**Идея**: Контракт ядерного ресурса (`_contract.yaml`) хранится **внутри** самого ресурса (`nuclear/{resource}/_contract.yaml`), а не централизованно в `_system/schema.yaml`.

**Механизм**:
- `_system/nuclear_registry.yaml` содержит только **указатели** (pointers) на зарегистрированные ядерные ресурсы.
- При копировании `nuclear/engineering/` в другой проект: копируется папка + `_contract.yaml`. В новом проекте достаточно добавить путь в `_system/nuclear_registry.yaml`.

**Плюсы**:
- ✅ **Портативность**: Ресурс самодостаточен. Контракт путешествует с данными.
- ✅ **Делегирование**: Владелец ресурса управляет его схемой, не трогая глобальную `schema.yaml`.
- ✅ **Версионирование**: Каждый ресурс эволюционирует в своём темпе.

**Риски**:
- ❌ **Консистентность**: Как обеспечить, чтобы локальный контракт не противоречил глобальным правилам безопасности (`_system/security_policy.yaml`)?
- ❌ **Связность**: Как ссылаться между ресурсами? Ссылка `[[id]]` может указывать на ресурс, которого нет в текущем проекте.

### 2. Связи между ядерными ресурсами
**Идея**: Ядерные ресурсы могут зависеть друг от друга.

**Механизм**:
- В `_contract.yaml` ресурса есть секция `dependencies`: список `nuclear_id`, которые требуются для работы.
- В `_system/nuclear_registry.yaml` есть секция `active_dependencies`, разрешающая конкретные связи.
- При импорте ресурса система проверяет: доступны ли зависимости? Если нет — ресурс помечается как `partial` или `broken_link`.

**Плюсы**:
- ✅ **Явность**: Зависимости задокументированы.
- ✅ **Безопасность**: Система блокирует доступ к ресурсу, если его зависимости не удовлетворены.

### 3. Изоляция памяти агентов
**Идея**: Память агентов — это не «принципы проекта», это динамическое состояние. Нужно отделить `context/project/` (статика) от `context/memory/agents/` (динамика).

**Структура**:
```
context/
├── principles/         # Статика: принципы, ценности, гайдлайны
├── glossary/           # Статика: термины
├── memory/             # Динамика
│   └── agents/         # Память конкретных агентов
│       ├── {agent_id}/
│       │   ├── short_term/  # Сессии (удаляются/архивируются)
│       │   ├── long_term/   # Усвоенные знания (версионируются)
│       │   └── _meta.yaml
│       └── shared_pool/     # Общая «коллективная память» (опционально)
```

**Плюсы**:
- ✅ **Гигиена**: Принципы проекта не смешиваются с «мыслями» агента.
- ✅ **Безопасность**: Память агентов можно шифровать/чистить отдельно, не трогая `principles/`.
- ✅ **Экспорт**: При передаче проекта новому агенту можно скопировать `principles/`, но оставить старую `memory/` (или загрузить новую).

### 4. Зоны тьмы (Dark Zones)
**Идея**: Явные ограничения для агентов, чтобы не тратить токены на лишнее и не галлюцинировать на нерелевантных данных.

**Типы Зон**:
- **Read-Blind**: Агент физически не видит содержимое (например, `nuclear/legal/contracts/` для агента-билдера). Возвращается пустота или ошибка `AccessDenied`.
- **Write-Prohibited**: Агент видит, но не может писать (например, `context/principles/`).
- **Token-Economic Block**: Агент *может* прочитать, но система предупреждает: «Этот файл весит 50К токенов, вы уверены?». Используется для `archive/` или огромных логов.

**Механизм**:
- Определяется в `_contract.yaml` ресурса или глобально в `_system/security_policy.yaml`.
- Пример:
  ```yaml
  dark_zones:
    - path: "archive/**"
      type: token_economic_block
      reason: "Архив не нужен для текущей задачи, экономит токены."
    - path: "nuclear/legal/**"
      type: read_blind
      roles_allowed: [legal_counsel]
  ```

**Плюсы**:
- ✅ **Экономия**: Экономия токенов и контекстного окна.
- ✅ **Фокус**: Агент концентрируется только на релевантном.
- ✅ **Безопасность**: Предотвращение утечки чувствительных данных в промпт.

## Предложение: Обновлённая структура `vault/`

```
vault/
├── _system/                      # Глобальные правила и реестры
│   ├── manifest.yaml
│   ├── security_policy.yaml      # Глобальные Dark Zones, шифрование
│   ├── nuclear_registry.yaml     # Указатели на ядерные ресурсы + зависимости
│   └── schema.yaml               # Базовые типы (только для системных ресурсов)
├── context/                      # Системные знания и память
│   ├── principles/               # Статика: принципы проекта
│   ├── glossary/                 # Статика: термины
│   └── memory/                   # Динамика: память агентов
│       └── agents/{agent_id}/
├── nuclear/                      # Ядерные ресурсы (динамические)
│   └── {resource_id}/
│       ├── _contract.yaml        # Локальный контракт ресурса
│       ├── dependencies.yaml     # Зависимости от других ядерных ресурсов
│       └── ...                   # Данные ресурса
├── workspace/                    # Активная работа
└── archive/                      # Архив (Dark Zone для экономии токенов)
```

## Вердикт анализа
Модель с локальными контрактами повышает **портативность** ядерных ресурсов, но усложняет **валидацию**. Система должна уметь проверять локальные контракты на соответствие глобальным правилам безопасности. Зоны тьмы — критически важны для масштабируемости и экономии. Разделение памяти — необходимо для гигиены и безопасности.
