# Анализ: Двухуровневая модель ресурсов `vault/` (System vs Nuclear)

## Тезис
`vault/` состоит из двух фундаментальных слоёв:
1.  **Системные ресурсы** (обязательные, универсальные): `_system/`, `context/`, `meta/`.
2.  **Ядерные ресурсы** (динамические, проектно-зависимые): `nuclear/`.

Эта модель обеспечивает баланс между **стандартизацией** (системный слой) и **гибкостью** (ядерный слой), позволяя адаптировать `vault/` под любой проект — от личной записной книжки до энтерпрайз-разработки.

---

## 1. Системные ресурсы (обязательные для любого `vault/`)

### `_system/` — Контракты и правила
| Файл | Назначение | Обязательно |
|------|------------|-------------|
| `manifest.yaml` | Манифест ресурса: версия схемы, название, поддерживаемые версии | ✅ Да |
| `schema.yaml` | Определение типов артефактов, состояний, связей, прав доступа | ✅ Да |
| `acl.yaml` | Детальные правила контроля доступа по ролям/агентам | ⚠️ Опционально (по умолчанию из `schema.yaml`) |
| `indexing.yaml` | Правила индексации: что включать, как парсить, кэшировать | ⚠️ Опционально |
| `migrations/` | Скрипты миграции между версиями схемы | ⚠️ Опционально |

### `context/` — Принципы, память агентов, общие знания
| Подпапка | Назначение | Примечание |
|----------|------------|------------|
| `principles/` | Архитектурные принципы, методология, гайдлайны | Универсальные для проекта |
| `agents/` | **Память агентов**: структурированные сессии, долгосрочные знания, контекст-паки | Проектно-зависимая структура, определяется в `schema.yaml` |
| `glossary/` | Глоссарий терминов, онтологии, таксономии | Кросс-доменные знания |
| `templates/` | Шаблоны артефактов по типам | Генерируются по `schema.yaml` |
| `history/` | Хронология ключевых решений, изменений, миграций | Аудит и онбординг |

**Агентская память в `context/agents/`**:
```
context/agents/
├── {agent_id}/
│   ├── short_term/           # Текущая сессия: черновики, промежуточные выводы
│   ├── long_term/            # Усвоенные знания: паттерны, предпочтения, ограничения
│   ├── session_summaries/    # Сжатые итоги сессий (для быстрого восстановления контекста)
│   └── _meta.yaml            # Метаданные агента: роль, права, история активаций
```
- Структура памяти конфигурируется в `_system/schema.yaml` (поле `agent_memory_schema`).
- Память привязана к проекту: при экспорте `vault/` память агентов экспортируется вместе с ним.
- Жизненный цикл: `active → summarized → archived`. Устаревшая память архивируется, но не удаляется.

### `meta/` — Операционные метаданные и аудит
| Подпапка | Назначение |
|----------|------------|
| `audit/` | Логи изменений, доступов, валидаций (для безопасности и отладки) |
| `indexes/` | Кэши индексов (SQLite, векторные) — генерируются автоматически |
| `migrations/` | История применённых миграций схемы |
| `runtime/` | Временные данные: блокировки, очереди, статусы задач |

**Важно**: `meta/` не содержит доменных знаний. Это «движок» `vault/`, а не «топливо».

## 2. Ядерные ресурсы (динамические, проектно-зависимые)

### `nuclear/` — Пространства доменных знаний
```
nuclear/
├── engineering/          # Пример: разработка ПО
│   ├── requirements/
│   ├── architecture/
│   ├── delivery/
│   └── _contract.yaml    # Локальные дополнения к контракту (опционально)
├── research/             # Пример: научное исследование
│   ├── hypotheses/
│   ├── experiments/
│   └── findings/
├── business/             # Пример: бизнес-контекст
│   ├── strategy/
│   ├── metrics/
│   └── stakeholders/
└── {custom_domain}/      # Любой новый домен добавляется созданием папки + записью в schema.yaml
```

### Контракт ядерного ресурса
Каждый ядерный ресурс **должен** соответствовать контракту, определённому в `_system/schema.yaml`:

```yaml
# Пример фрагмента _system/schema.yaml
nuclear_resources:
  engineering:
    description: "Знания, связанные с разработкой ПО"
    required_types: [atom, decision, note]
    optional_types: [test_plan, api_contract, deployment_guide]
    acl:
      read: [architect, builder, auditor]
      write: [architect, builder]
      approve: [architect]
    memory_integration:
      agent_roles: [researcher, architect, builder]
      store_session_summaries: true
      vectorize_for_rag: true

  research:
    description: "Научные исследования, эксперименты"
    required_types: [research_log, inquiry, note]
    optional_types: [dataset_spec, ethics_review]
    acl:
      read: [researcher, architect, auditor]
      write: [researcher]
      approve: [researcher]  # Само-аппрув для research
```

**Правила**:
1.  Новый ядерный ресурс добавляется созданием папки в `nuclear/` **и** записью его контракта в `_system/schema.yaml`.
2.  Контракт определяет: допустимые типы артефактов, права доступа, интеграцию с агентской памятью.
3.  Ядерный ресурс может иметь локальный `_contract.yaml` для уточнения правил (не переопределения глобальных).
4.  Удаление ядерного ресурса требует архивации всех артефактов и обновления `schema.yaml`.

## 3. Адаптивность: от записной книжки до энтерпрайза

| Тип проекта | Системные ресурсы | Ядерные ресурсы | Конфигурация |
|-------------|------------------|-----------------|--------------|
| **Личная записная книжка** | `_system/`, `context/`, `meta/` | `nuclear/notes/` (один ресурс) | Минимум типов: `note`, `inquiry`. Нет ACL, простая схема. |
| **Исследовательский проект** | `_system/`, `context/`, `meta/` | `nuclear/research/`, `nuclear/data/` | Типы: `research_log`, `dataset_spec`, `ethics_review`. ACL: исследователь = аппрувер. |
| **Software development** | `_system/`, `context/`, `meta/` | `nuclear/engineering/`, `nuclear/product/`, `nuclear/ops/` | Полная схема: `atom`, `adr`, `test_plan`, `deployment_guide`. Строгие ACL, интеграция с CI/CD. |
| **Юридический документ** | `_system/`, `context/`, `meta/` | `nuclear/legal/`, `nuclear/compliance/` | Типы: `clause`, `precedent`, `review_log`. ACL: юрист = единственный писатель. |

**Ключевое**: Системные ресурсы **всегда одинаковы**. Меняется только `_system/schema.yaml` и набор ядерных ресурсов.

## 4. Преимущества модели

| Преимущество | Как достигается |
|--------------|----------------|
| **Портативность** | Скопировать `vault/` → настроить `schema.yaml` → готово. Системный слой не требует изменений. |
| **Масштабируемость** | Новые домены добавляются без изменения кода ядра. Индексация шардируется по ядерным ресурсам. |
| **Изоляция ответственности** | Системные конфиги (`_system/`) не смешиваются с доменными данными (`nuclear/`). |
| **Агентская память как первый класс** | Память структурирована, версионируется, экспортируется вместе с проектом. |
| **Контрактная гарантия** | Каждый ядерный ресурс имеет явный контракт. Инструменты могут валидировать соответствие до записи. |
| **Эволюция без разрывов** | Изменение `schema.yaml` не ломает старые артефакты (поддержка нескольких версий типов). |

## 5. Риски и митигация

| Риск | Влияние | Митигация |
|------|---------|-----------|
| **Сложность миграции** | Существующие `vault/` требуют перестройки под новую модель | Предоставить скрипт `vault migrate to-system-nuclear` с бэкапом и валидацией |
| **Когнитивная нагрузка** | Новые пользователи могут запутаться в двухуровневой структуре | Создать `context/templates/vault_starter_guide.md` с примерами для разных типов проектов |
| **Дрейф контрактов** | Ядерный ресурс может отклониться от контракта, определённого в `schema.yaml` | Валидатор ядра проверяет соответствие при каждом коммите; алерт при расхождении |
| **Перегрузка агентской памяти** | Агенты могут накапливать избыточную память, замедляя поиск | Политика жизненного цикла: `active → summarized → archived`. Автоматическая компрессия старых сессий. |
| **Безопасность памяти агентов** | Память может содержать чувствительные данные | Шифрование `context/agents/` на уровне ФС; очистка памяти при завершении сессии (настраивается в `acl.yaml`). |

---

## 6. Вердикт

Двухуровневая модель **решает проблему адаптивности без потери стандартизации**. Системные ресурсы обеспечивают единый контракт и правила игры. Ядерные ресурсы позволяют гибко моделировать любой домен. Агентская память, интегрированная в `context/`, становится первым классом артефактом, а не побочным продуктом.

**Рекомендация**: Принять модель как целевую архитектуру `vault/`. Реализовать поэтапно:
1.  Добавить `_system/manifest.yaml` и `schema.yaml` (не ломая текущее).
2.  Реорганизовать `context/` для поддержки агентской памяти.
3.  Переименовать `delivery/`, `engineering/` и т.д. в `nuclear/`.
4.  Реализовать валидатор контрактов ядерных ресурсов.
