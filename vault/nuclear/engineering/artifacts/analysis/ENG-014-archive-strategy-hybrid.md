# Критический анализ: Гибридная модель архивации (Локальный + Глобальный + Метаданные)

## Прямой ответ
**Да. Нужны оба уровня: локальный архив внутри ядерного ресурса и глобальный системный архив. Но не как физическая дихотомия, а как логический континуум с чёткими правилами синхронизации.**

Архивация по умолчанию — **только метаданные** (`status: archived`). Это сохраняет все связи `[[id]]` без разрывов. При этом архивный контент автоматически попадает в **Зону тьмы** (агенты его игнорируют, экономя токены).

Физическое перемещение — **опциональный шаг оптимизации/комплаенса**. Локальный архив (`nuclear/{id}/_archive/`) сохраняет мобильность ресурса. Глобальный архив (`vault/archive/global/`) обеспечивает кросс-ресурсный аудит, retention-политики и очистку хранилища.

## 🔍 Критика экспертной панели (7 экспертов)

### 👩‍💻 Алиса (Системный архитектор)
**Вердикт: ✅ Гибридная модель решает парадокс портативности vs целостности**
- Копирование `nuclear/engineering/` должно включать его историю. Локальный `_archive/` даёт это «из коробки».
- Но без глобального индекса/архива система не масштабируется: 50 ресурсов = 50 разрозненных историй. Аудит превращается в квест.
- **Решение:** Локальный архив — *источник истины для ресурса*. Глобальный — *синхронизированный индекс + кэш для комплаенса*. Связи резолвятся через единый `id`-резолвер, который проверяет локальный путь → глобальный индекс → fallback.

### 🧠 Боб (Управление знаниями)
**Вердикт: ✅ Сохраняет эволюцию контекста, не теряя фокус**
- Черновики, отменённые спеки, устаревшие ADR — часть истории принятия решений. Удалять их = амнезия. Перемещать в глобальный архив = лишать ресурс контекста.
- Локальный `_archive/` сохраняет «почему мы так не сделали» прямо рядом с текущей работой.
- **Риск:** Ресурс «раздувается» старыми файлами. **Митигация:** `vault compact --nuclear engineering` → архивирует, сжимает, генерирует `archive_summary.md` для быстрого контекста без загрузки бинарников.

### 🔒 Кэрол (Security/DevOps)
**Вердикт: ⚖️ Локальный — для мобильности, Глобальный — для принудительного удержания**
- GDPR, HIPAA, SOX требуют централизованного управления retention. Если каждый ресурс сам архивирует/удаляет — комплаенс невозможен.
- **Решение:** Глобальный архив не копирует файлы, а хранит **зашифрованные хэши + метаданные + политики удаления**. Локальный ресурс владеет данными, глобальный — контролирует жизненный цикл.
- При копировании ресурса в другой проект: копируются файлы + `_archive/`. Глобальный архив нового проекта создаёт новый слепок, не наследуя чужие retention-правила.

### 🤖 Дэвид (AI/LLM)
**Вердикт: ✅ Тёмные архивы = спасение контекстного окна**
- `status: archived` → автоматическое добавление в `dark_zones` с `type: token_economic_block`.
- Агент не сканирует архивы по умолчанию. Только явный `--include-archive` или эскалация `INQ.HISTORICAL_CONTEXT`.
- Векторный индекс архивов строится отдельно (`archive_embeddings/`). Не смешивается с активным поиском.
- При копировании ресурса векторы локального архива копируются вместе с ним. Глобальный индекс обновляется лениво.

### 📊 Елена (Стратег)
**Вердикт: ✅ Профильная гибкость без over-engineering**
- Для `profile: minimal` → только метаданные. Физические архивы отключены.
- Для `profile: standard` → локальный `_archive/` включён, глобальный — только индекс.
- Для `profile: enterprise` → полная гибридная модель с синхронизацией, шифрованием и автоматическим retention.
- Не навязывать сложность. Дать конфиг `archive_policy: [metadata_only, local_physical, hybrid_sync]`.

### ⚡ Фрэнк (Производительность)
**Вердикт: ✅ CAS + Логические указатели решают проблему дублирования**
- Физическое копирование архивов при `cp -r nuclear/A/` = раздувание диска. 
- **Решение:** Архивы хранятся в Content-Addressable Storage (CAS) по `content_hash`. Локальный `_archive/` хранит только указатели (`refs.yaml`). Глобальный архив — общий пул блоков.
- При копировании ресурса указатели копируются мгновенно. Реальные данные подтягиваются лениво или дедуплицируются при первом обращении.
- Поиск по архивам: сначала локальный индекс, затем глобальный. Кэш на уровне L1.

### ⚖️ Грейс (Этика/Право)
**Вердикт: 🟡 Архив — это не свалка, это регулируемое хранилище**
- Черновики можно удалять физически. Решения, контракты, ADR — никогда. 
- Разделение: `nuclear/{id}/_archive/drafts/` (можно удалять по TTL) vs `nuclear/{id}/_archive/decisions/` (immutable).
- Глобальный архив должен вести `audit_trail` всех физических перемещений. Удаление из локального архива ≠ удаление из глобального следа.

## 🏗 Предлагаемая архитектура: Гибридная модель

```yaml
# vault/_system/archive_policy.yaml
archive_defaults:
  metadata_only: true            # По умолчанию: только статус + dark_zone
  physical_mode: local           # local | global | hybrid_cas
  dark_zone_auto: true           # Архив = автоматически в Зону тьмы
  agent_access: restricted       # requires_approval | token_budget_deduction

ttl_policies:
  drafts: 90d                    # Физическое удаление черновиков
  rejected_proposals: 180d       
  approved_decisions: immutable  # Никогда не удалять
```

### 1. Метаданные (L1 — основа)
```yaml
# В любом артефакте
status: archived
archived_at: 2026-04-11
archive_reason: superseded_by_v2
supersedes_by: DEL.FEAT.AUTH.v2
# Автоматически: попадает в dark_zones, исключается из активного индекса
```

### 2. Локальный архив ядерного ресурса (L3 — мобильность)
```
nuclear/engineering/
├── _contract.yaml
├── artifacts/...
└── _archive/
    ├── index.sqlite          # Локальный индекс с id → path/status
    ├── refs.yaml             # CAS-указатели на физические блоки
    ├── drafts/               # TTL 90d, автоочистка
    └── decisions/            # Immutable, хранится с ресурсом
```
- Копирование `nuclear/engineering/` → полный контекст + история.
- Физические файлы могут быть символическими ссылками на CAS или реальными копиями (зависит от `physical_mode`).

### 3. Глобальный архив (L1 — комплаенс + аудит)
```
vault/archive/global/
├── master_index.sqlite       # Единый граф всех архивных id
├── cas_store/                # Дедуплицированное хранилище по hash
├── retention_engine/         # Политики удаления, юридические холды
└── audit_log/                # Кто, когда, почему архивировал/удалял
```
- Не хранит дубликаты. Хранит `resource_id:hash:policy`.
- При удалении локального ресурса глобальный архив сохраняет слепок, если `retention_policy` требует.

## 🔗 Как сохраняются связи и мобильность

| Действие | Что происходит со связями | Что происходит с физикой |
|----------|--------------------------|--------------------------|
| `status: archived` (метаданные) | Все `[[id]]` работают. Резолвер помечает узел `archived`. | Файлы остаются на месте. Попадают в `dark_zones`. |
| `vault archive --physical local` | Связи обновляются на CAS-указатели. `id` не меняется. | Файлы перемещаются в `nuclear/{id}/_archive/`. Активная папка чистится. |
| Копирование `nuclear/{id}/` в новый проект | Все ссылки валидны внутри ресурса. Внешние `[[id]]` → `broken_link` (ожидаемо). | Копируются `_archive/refs.yaml` + CAS-блоки (или скачиваются лениво). |
| `vault sync-archive` (L1↔L3) | Глобальный индекс обновляется. Конфликты retention разрешаются по приоритету. | CAS дедуплицируется. Удалённые локально файлы сохраняются в глобальном холде (если требуется). |

**Резолвер `[[id]]` (алгоритм):**
1. Проверяет активный `nuclear/{current_id}/artifacts/`
2. Если не найден → `nuclear/{current_id}/_archive/refs.yaml`
3. Если не найден → глобальный `master_index.sqlite`
4. Если не найден → `AccessDenied` или `broken_link` (в зависимости от `dark_zone`)

## 🎯 Матрица принятия решений (конфигурация)

| Сценарий | `metadata_only` | `physical_mode` | `dark_zone_auto` | Поведение |
|----------|----------------|-----------------|------------------|-----------|
| Личная записная книжка | ✅ | none | ✅ | Только статус. Файлы не двигаются. Агенты игнорируют. |
| Команда 5-10 человек | ✅ | local | ✅ | Черновики чистятся по TTL. Решения остаются в `_archive/`. Ресурс мобилен. |
| Enterprise / Regulated | ✅ | hybrid_cas | ✅ + audit | Глобальный retention-холд. Локальные копии — кэш. Полный аудит. |
| AI-агентный конвейер | ✅ | local + vector_isolation | ✅ | Архивы исключаются из векторного индекса. Экономия токенов >70%. |

## ⚖️ Вердикт Мастера

**Уважаемый Со-архитектор,**

Ваш инстинкт верен: **портативность ядерного ресурса не должна приноситься в жертву централизации**. Но физическое перемещение в *только* глобальный архив — ошибка. Физическое перемещение в *только* локальный — риск для комплаенса.

### Принимается:
✅ **Гибридная модель с приоритетом метаданных.** `status: archived` — первично. Физика — оптимизация.
✅ **Локальный `_archive/` внутри ядерного ресурса.** Гарантирует 100% мобильность и контекстную связность.
✅ **Автоматическое включение архивов в Зоны тьмы.** Агенты не тратят токены на историю по умолчанию.
✅ **Глобальный архив как комплаенс-слой, не как хранилище файлов.** Хранит индексы, политики, CAS-хэши, аудит-трейлы.

### Отклоняется:
❌ **Обязательное физическое перемещение в глобальный архив** — ломает мобильность, создаёт зависимость от сети/доступа.
❌ **Удаление архивных файлов без глобального следа** — нарушает регуляторику и принцип «архив ≠ мусорка».

### Архитектурное правило:
> **«Архив — это состояние, не место. Локальный `_archive/` сохраняет контекст ресурса. Глобальный `archive/` сохраняет целостность системы. Связи резолвятся по `id`. Физика следует за логикой, не наоборот.»**
