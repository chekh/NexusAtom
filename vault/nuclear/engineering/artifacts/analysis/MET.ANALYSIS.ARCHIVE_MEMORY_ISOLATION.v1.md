# Критический анализ: Децентрализация архива и памяти агентов по ядерным ресурсам

## Прямой ответ
**Нет. Полная физическая децентрализация архива и памяти агентов по ядерным ресурсам — это архитектурный анти-паттерн для SSOT.** Она разрушает граф знаний, фрагментирует индекс, усложняет аудит и ломает ментальную модель агента. Однако **логическое разделение (scoping) + опциональная физическая изоляция для комплаенса** — необходимы.

Ниже — жёсткая критика от экспертной панели с опорой на установленные контракты уровней.

---

## 🔍 Критика экспертной панели

### 👩‍💻 Алиса (Системный архитектор)
**Вердикт: ❌ Нарушает контракт L1**
- L1 — единый граф знаний. Архив и память — его фундамент. Разнесение их по `nuclear/` превращает `vault/` в набор изолированных силосов.
- Кросс-ресурсные связи (`[[id]]`) ломаются, когда архивные узлы или память агентов недоступны вне контекста ресурса.
- Миграция агента между ресурсами становится кошмаром: как перенести `long_term/` из `nuclear/A/` в `nuclear/B/`? Придётся писать кастомные скрипты миграции памяти.
- **Рекомендация:** Сохранить физическую централизацию в L1. Изоляцию реализовывать через **логические партиции** (namespaces, ACL, теги), а не через копирование папок.

### 🧠 Боб (Управление знаниями)
**Вердикт: ❌ Фрагментирует обучение и онбординг**
- Память агента — это его накопленный опыт. Опыт должен быть **агент-центричным**, а не ресурс-центричным. Если агент работал в `engineering` и `research`, его инсайты теряют связность при разделении памяти.
- Архив — это организационная память. Команда должна видеть полную историю решений, а не собирать мозаику из 10 разрозненных архивов.
- Когнитивная нагрузка для новичков: «Где искать старые спеки по авторизации? В архиве `nuclear/security/` или в `nuclear/engineering/`? А если они связаны?»
- **Рекомендация:** Архив — единый индекс с фильтрацией по `nuclear_id`. Память — привязана к `agent_id`, но с возможностью загрузки **контекстно-скорированных** срезов.

### 🔒 Кэрол (Security/DevOps)
**Вердикт: ⚖️ Поддерживаю изоляцию только для комплаенса, не как дефолт**
- Если `nuclear/legal/` содержит GDPR/PCI-DSS данные, держать его архив и память агентов в общей папке = риск утечки при ошибке конфигурации.
- Но физическое разделение по умолчанию = ад для аудита. Как проверить, что агент не скопировал чувствительные данные в `nuclear/public/`? Придётся сканировать N архивов.
- **Рекомендация:** В `_contract.yaml` ядерного ресурса добавить флаг `isolation_mode: [standard, compliant, air_gapped]`. При `compliant` — ядро физически выносит архив/память в зашифрованный том, но оставляет **логический указатель** в центральном индексе для аудита.

### 🤖 Дэвид (AI/LLM Engineer)
**Вердикт: ❌ Убивает RAG и кросс-доменный reasoning**
- Векторный индекс по 10 разрозненным архивам/памятям = падение точности поиска на 40-60%. Модель не увидит связи между `engineering/spec_v1` (архив) и `legal/contract_v2` (архив).
- Контекстное окно: загрузка памяти из 3 разных `nuclear/` папок = дублирование системных промптов, перерасход токенов, конфликт инструкций.
- Агенту нужен **единый пул знаний с динамическим скорингом релевантности**, а не 10 изолированных баз.
- **Рекомендация:** Централизованный векторный индекс с метаданными `scope: nuclear_id`. При загрузке контекста ядро применяет `context_budget` и ранжирует результаты по активному ресурсу, не вырезая остальное физически.

### 📊 Елена (Стратег)
**Вердикт: ❌ Over-engineering для 95% проектов**
- Для стартапа или команды 5 человек разделение архива/памяти по ресурсам = бюрократия, а не безопасность.
- Онбординг нового агента/человека превращается в квест: «Настрой доступ к архиву A, подключи память B, проверь зависимости C...»
- **Рекомендация:** Профиль `minimal`/`standard` — единый архив и память. Профиль `enterprise` — опциональная изоляция через `isolation_mode`. Не навязывать сложность по умолчанию.

### ⚡ Фрэнк (Производительность)
**Вердикт: ❌ Индексный ад**
- Вместо одного SQLite/Vector DB — N баз. Кэш не переиспользуется. Запросы `search --cross-resource` требуют `UNION ALL` с падением производительности.
- Инкрементальное обновление индекса при коммите усложняется: нужно знать, в каком `nuclear/` архиве/памяти изменились файлы.
- **Рекомендация:** Единый индекс с партиционированием по `nuclear_id` и `agent_id`. Физически один файл/том, логически — изолированные таблицы/коллекции. Производительность растёт, сложность падает.

### ⚖️ Грейс (Этика/Право)
**Вердикт: 🟡 Память — нет, Архив — да (условно)**
- Архив может содержать данные с разными политиками удержания (retention policies). GDPR требует удаления через 2 года, инженерные спеки — хранить 10 лет. Физическое разделение упрощает compliance.
- Но память агента — это синтез знаний, не сырые данные. Её разделение нарушает принцип минимизации и усложняет «право на забвение» (придётся чистить N папок).
- **Рекомендация:** Архив — логически единый, но с тегами `retention_policy` и опциональным шифрованием на уровне ресурса. Память агентов — всегда централизована, с автоматической ротацией и шифрованием ключом агента.

## ⚖️ Вердикт Мастера

**Уважаемый Со-архитектор,**

Ваш вопрос точен, но решение лежит в плоскости **логического разделения, а не физической фрагментации**.

### Что отклоняем:
❌ **Физическое копирование `archive/` и `memory/` внутрь `nuclear/{id}/`**. Это разрушает SSOT, ломает граф, убивает RAG, создаёт индексный хаос.

### Что принимаем:
✅ **Логическое скорирование (Logical Scoping)**: Архив и память остаются в L1, но индексируются с метаданными `scope: nuclear_id`.
✅ **Опциональная физическая изоляция для комплаенса**: Через `isolation_mode` в `_contract.yaml`. Ядро автоматически выносит данные в зашифрованный том, но сохраняет логический указатель для аудита.
✅ **Агент-центричная память, контекстно-скорированная**: Память привязана к `agent_id`, но при загрузке контекста ядро ранжирует её по активному `nuclear_id`, применяя `context_budget`.

### Архитектурное правило:
> **«L1 хранит истину единой. L3 декларирует требования к изоляции. L2 управляет доступом. Физическая фрагментация допускается только по явному комплаенс-триггеру, не по умолчанию.»**

## 🛠 Рекомендуемая реализация

### 1. Архив (L1)
```
vault/archive/
├── index.sqlite              # Единый индекс с полями: nuclear_id, retention_policy, encryption_status
├── logical/                  # Логические ссылки на физические данные
│   ├── nuclear_engineering/
│   └── nuclear_legal/
└── physical/                 # Физическое хранение (по умолчанию единое)
    └── {hash}/               # Контент-адресуемое хранилище (CAS)
```
- `vault archive query --nuclear engineering` → быстрый фильтр по индексу.
- `isolation_mode: compliant` → шифрование на уровне файлов, отдельный ключ.

### 2. Память агентов (L1)
```
vault/context/memory/agents/{agent_id}/
├── long_term/
│   ├── embeddings/           # Векторный индекс (единый, с тегом agent_id + nuclear_scope)
│   └── summaries/            # Сжатые выводы сессий
├── _meta.yaml                # role, active_scopes: [engineering, research], memory_budget: 50000
└── access_control.yaml       # Какие ядерные ресурсы могут читать/писать в эту память
```
- Агент загружает память через `agent memory-load --scope engineering`.
- Ядро применяет `context_budget`, отсекая нерелевантные части.

### 3. Контракт ядерного ресурса (L3)
```yaml
# nuclear/legal/_contract.yaml
isolation_mode: compliant
data_retention: {default: 7y, gdpr: 2y}
dark_zones:
  - path: "contracts/"
    type: read_blind
    reason: "GDPR Article 17"
memory_policy: {encrypt: true, share_scope: [legal_counsel]}
```

## ✅ Итог
Архив и память агентов **должны оставаться в L1** для сохранения SSOT, производительности и кросс-доменного reasoning. Изоляция достигается через **логическое скорирование, метаданные индекса и опциональное шифрование**, а не через копирование папок в `nuclear/`. Это сохраняет портативность, не жертвуя безопасностью и масштабируемостью.
