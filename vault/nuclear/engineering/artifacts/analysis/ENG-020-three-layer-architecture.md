# Анализ: Трёхуровневая архитектура NexusAtom (Знания → Процессы → Доставка)

## Тезис
NexusAtom не является монолитом. Это **композитная система**, состоящая из трёх логически независимых, но функционально связанных уровней:
1.  **Уровень 1: Ядро знаний (Vault Core)** — портативная база данных текстовых знаний, SSOT для агентов и людей.
2.  **Уровень 2: Управление процессами (Orchestration Layer)** — правила, методологии, FSM, маршрутизация задач поверх знаний.
3.  **Уровень 3: Доставка ценности (Delivery Layer)** — систематизация артефактов исполнения (specs, ADR, test-plans, code manifests).

Каждый уровень имеет чёткий контракт, зону ответственности и механизмы взаимодействия. **Зоны тьмы** интегрированы в единую схему: определяются на Уровне 3, управляются на Уровне 2, применяются на Уровне 1.

---

## 🟦 Уровень 1: Ядро знаний (Vault Core)
**Назначение:** Универсальный, агентно-ориентированный контейнер для централизации, валидации, индексации и безопасного доступа к текстовым знаниям.

**Компоненты в `vault/`:**
```
vault/
├── _system/              # Глобальные контракты, безопасность, реестры
├── context/principles/   # Статические принципы, ценности, глоссарий
├── context/memory/       # Динамическая память агентов (изолированная)
├── meta/                 # Аудит, индексы, миграции, кэши
├── workspace/            # Активная работа (inbox, drafts)
└── archive/              # Логический архив
```

**Ключевые механизмы:**
- ✅ **Само-описываемость:** `manifest.yaml` + `nuclear_registry.yaml`.
- ✅ **Каскадная валидация:** Локальные контракты ядерных ресурсов проверяются на соответствие глобальной политике.
- ✅ **Индексация на уровне ядра:** Векторный поиск, граф связей, семантический резолвинг `[[id]]`.
- ✅ **Изоляция памяти:** `context/memory/agents/` шифруется, версионируется, управляется отдельно.
- ✅ **Зоны тьмы (применение):** Исключение путей из индекса, контроль доступа, экономия токенов.

**Выход уровня:** Валидированный, индексированный, безопасный граф знаний.

---

## 🟨 Уровень 2: Управление процессами (Orchestration Layer)
**Назначение:** Организация *того, как* знания создаются, ревьюятся, утверждаются и перемещаются между состояниями. Не хранит сами артефакты доставки, а определяет **правила игры**.

**Компоненты в `vault/`:**
```
context/workflows/
├── methodologies/        # Agile, Research, Kanban, Stage-Gate, Waterfall
├── fsm-definitions/      # Машины состояний для типов артефактов
├── routing-rules/        # Маршруты ревью, аппрува, эскалации
└── governance-policies/  # Политики качества, compliance, этики
```

**Ключевые механизмы:**
- ✅ **Методологическая агностичность:** Процесс не диктуется ядром, а конфигурируется через `methodologies/`.
- ✅ **FSM + Гейты:** Переходы статусов (`draft → rfc → approved`) управляются правилами из `fsm-definitions/`.
- ✅ **Маршрутизация:** Задачи направляются нужным ролям (`architect`, `legal`, `product`) через `routing-rules/`.
- ✅ **Зоны тьмы (управление):** Политики аудита, требования обоснования `reason`, логирование попыток обхода.

**Выход уровня:** Детерминированные рабочие потоки, статус-машины, цепочки согласования.

---

## 🟥 Уровень 3: Доставка ценности (Delivery Layer)
**Назначение:** Конкретная систематизация артефактов, которые производят измеримую ценность: код, отчёты, юридические документы, исследовательские выводы.

**Компоненты в `vault/`:**
```
nuclear/
└── {resource_id}/
    ├── _contract.yaml        # Локальный контракт ресурса (типы, права, dark_zones)
    ├── dependencies.yaml     # Зависимости от других ядерных ресурсов
    └── artifacts/            # Контейнеры доставки (atom, research_log, legal_brief, etc.)
        └── {artifact_id}/
            ├── spec.md
            ├── adr.md
            ├── test_plan.md
            └── manifest.yaml
```

**Ключевые механизмы:**
- ✅ **Контрактная изоляция:** Каждый ядерный ресурс сам определяет свои артефакты и правила.
- ✅ **Зоны тьмы (определение):** `dark_zones` задаются в `_contract.yaml` на уровне ресурса.
- ✅ **Манифест-трекинг:** Связь абстрактных спеков с физическими файлами в репозитории (`manifest.yaml`).
- ✅ **Доставка:** CI/CD, генерация кода, публикация отчётов, деплой.

**Выход уровня:** Готовый продукт, код, отчёт, юридический акт, научная публикация.

## 🔗 Интеграция Зон тьмы в единую схему

Зоны тьмы не принадлежат одному уровню. Они **сквозной механизм безопасности**, связывающий все три:

| Уровень | Роль в Зонах тьмы | Пример |
|---------|------------------|--------|
| **L3: Delivery** | **Определяет** зоны в `_contract.yaml` ядерного ресурса | `dark_zones: [{path: "legal/contracts/", type: read_blind, reason: "GDPR"}]` |
| **L2: Orchestration** | **Управляет** аудитом, обоснованием, эскалацией при обходе | `governance/dark_zone_audit.yaml`: требование `requires_approval: true` для снятия зоны |
| **L1: Core** | **Применяет** на уровне индексации и доступа | Индексатор исключает `legal/contracts/` из векторного поиска. Резолвер возвращает `AccessDenied` |

**Поток валидации Зон тьмы:**
1.  Ресурс регистрируется в `_system/nuclear_registry.yaml`.
2.  L1 проверяет: `_contract.yaml` не противоречит `_system/security_policy.yaml`.
3.  L2 проверяет: все `dark_zones` имеют `reason`, логируются, соответствуют `governance-policies/`.
4.  L1 применяет: пути удаляются из векторного индекса, доступ блокируется на уровне API.

---

## 🌊 Пример сквозного потока (Исследование → Продукт)

```
1. Идея (L1) → Захват в workspace/inbox/ → статус raw
2. Маршрутизация (L2) → workflow: research_methodology → FSM: raw → proposed
3. Контейнер (L3) → Создание nuclear/research/ с _contract.yaml + dark_zones (если есть чувствительные данные)
4. Исполнение (L3) → Агенты пишут findings, adr, test_plan. Ядро (L1) валидирует контракт, индексирует без Зон тьмы.
5. Переход к доставке (L2) → workflow: product_dev → FSM: approved → in_progress
6. Создание атома (L3) → nuclear/engineering/delivery/atom/auth_v2/ → manifest.yaml → код → CI → release
```

## ⚠️ Критический анализ и риски

| Риск | Влияние | Митигация |
|------|---------|-----------|
| **Размытие границ уровней** | Процессы (L2) смешиваются с данными (L3) или знаниями (L1) | Строгая валидация путей: `context/workflows/` ≠ `nuclear/`, `_system/` ≠ `context/memory/` |
| **Каскадная валидация bottleneck** | Проверка всех 3 уровней при каждом изменении замедляет работу | Инкрементальная валидация + кэш `contract_cache.sqlite` + триггеры только на затронутые уровни |
| **Конфликт Зон тьмы** | L3 определяет зону, L2 требует её снятия для аудита, L1 блокирует доступ | Приоритет: L1 (безопасность) > L2 (аудит с аппрувом) > L3 (локальные правила). Явный `override` требует мульти-сигнатуры |
| **Избыточность для малых проектов** | Три уровня кажутся тяжелыми для простой записной книжки | Профиль `minimal`: L1 только, L2 = базовый FSM, L3 = 1 ядерный ресурс без зависимостей |

## ✅ Вердикт

Трёхуровневая модель **логически непротиворечива и архитектурно устойчива**:
- **L1** обеспечивает *портативность и безопасность* знаний.
- **L2** обеспечивает *детерминизм и гибкость* процессов.
- **L3** обеспечивает *измеримость и доставку* ценности.

Зоны тьмы интегрированы сквозным образом, что исключает «утечку» контекста и обеспечивает экономию токенов без потери аудируемости.

**Рекомендация**: Принять модель как целевую архитектуру NexusAtom. Реализовать поэтапно: сначала L1 (ядро знаний), затем L3 (ядерные ресурсы + контракты), затем L2 (процессы + маршрутизация).
