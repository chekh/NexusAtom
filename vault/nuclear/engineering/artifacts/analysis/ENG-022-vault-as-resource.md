# Анализ: `vault/` как независимый ресурс знаний

## Тезис
`vault/` — не просто папка в репозитории. Это **самостоятельный, портативный, само-описываемый контейнер знаний** (Independent Knowledge Resource — IKR). Его структура должна выводиться из этого свойства, а не из особенностей текущего проекта.

## Критический анализ текущей структуры

### ✅ Сильные стороны (для этого проекта)
| Аспект | Почему работает |
|--------|----------------|
| Разделение `context/meta/delivery` | Чёткие границы ответственности для разработки NexusAtom |
| Префиксная система `id` | Мгновенная идентификация типа артефакта |
| Логическая архивация | Сохраняет историю, не ломает ссылки |
| `registry.yaml` | Динамическая конфигурация типов |

### ❌ Слабые стороны (для портативности)
| Проблема | Влияние на другие проекты |
|----------|---------------------------|
| **Жёсткие папки** (`context/`, `meta/`, `delivery/`) | Не подходят для чистого research-проекта (нет `delivery`), или для монолита (нет разделения) |
| **Предположение о домене** | Структура заточена под software development. Не адаптируется под юридический документ, научную статью, маркетинговую кампанию |
| **Смешение системного и доменного** | `meta/registry.yaml` — системный конфиг, но лежит рядом с доменными артефактами. Усложняет перенос |
| **Отсутствие явного манифеста ресурса** | Нет файла, который говорит: «Это `vault/` версии 2.0, используй схему `schema.yaml` для парсинга» |
| **Зависимость от инструмента** | Структура предполагает наличие NexusAtom CLI. Без него навигация затруднена |

## Логика структурирования от «ресурса»

Если `vault/` — независимый ресурс, то:

1.  **Он должен быть само-описываемым**. Внутри должен быть манифест, определяющий:
    - Версию схемы ресурса (`vault_schema_version: 2.0`)
    - Доступные типы артефактов (заменяет `registry.yaml`)
    - Правила именования, валидации, индексации
    - Систему прав доступа (ACL) по умолчанию

2.  **Он должен быть домен-агностичным**. Структура папок не должна диктовать, *о чём* знания. Она должна обеспечивать *как* они организованы, найдены и связаны.

3.  **Он должен поддерживать логические пространства (Namespaces)**. Вместо жёстких `context/meta/delivery` — гибкие `namespaces/`, которые конфигурируются под проект.

4.  **Он должен иметь чёткий контракт на вход/выход**. Как импортировать знания? Как экспортировать? Как синхронизировать с кодом?

## Предложение: Универсальный скелет `vault/` (v2)

```
vault/
├── _system/                          # Системные файлы ресурса (не доменные)
│   ├── manifest.yaml                 # Манифест ресурса: версия, схема, правила
│   ├── schema.yaml                   # Определение типов, состояний, связей (заменяет registry.yaml)
│   ├── acl.yaml                      # Правила доступа по ролям/агентам
│   ├── indexing.yaml                 # Правила индексации: что включать, как парсить
│   └── migrations/                   # Скрипты миграции между версиями схемы
├── namespaces/                       # Логические пространства знаний (конфигурируемые)
│   ├── engineering/                  # Пример: для software project
│   │   ├── requirements/
│   │   ├── architecture/
│   │   └── delivery/
│   ├── business/                     # Пример: для бизнес-контекста
│   │   ├── strategy/
│   │   ├── metrics/
│   │   └── stakeholders/
│   └── research/                     # Пример: для научного проекта
│       ├── hypotheses/
│       ├── experiments/
│       └── findings/
│   # Новые пространства добавляются созданием папки + записью в _system/schema.yaml
├── workspace/                        # Активная работа: черновики, текущие атомы, WIP
│   ├── inbox/                        # Быстрый захват (заменяет funnel/)
│   ├── drafts/                       # Черновики артефактов
│   └── current/                      # Ссылки на активные атомы из любых namespaces
├── archive/                          # Логический архив: артефакты со status: archived
│   # Физически файлы остаются на местах, но индекс помечает их как архивные
│   # Опционально: физическое перемещение через `vault archive --physical`
└── shared/                           # Общие, кросс-доменные знания (глоссарий, онтологии, шаблоны)
    ├── glossary/
    ├── templates/
    └── ontologies/
```

### Ключевые изменения

| Было | Стало | Почему |
|------|-------|--------|
| `meta/registry.yaml` | `_system/schema.yaml` | Явное разделение: системная схема ≠ доменные данные |
| Жёсткие папки `context/meta/delivery` | Гибкие `namespaces/` | Адаптация под любой домен через конфигурацию |
| `funnel/` в корне | `workspace/inbox/` | Логическая группировка: всё активное — в `workspace/` |
| Нет манифеста ресурса | `_system/manifest.yaml` | Само-описываемость: версия, схема, правила |
| Архив = статус в файле | `archive/` + индекс | Явное пространство для архивных данных, упрощает фильтрацию |

### Преимущества нового подхода

1.  **Портативность**: Скопировать `vault/` в новый проект → настроить `namespaces/` в `schema.yaml` → готово.
2.  **Масштабируемость**: Новые домены добавляются созданием папки в `namespaces/`, не изменением корневой структуры.
3.  **Изоляция**: Системные конфиги (`_system/`) не смешиваются с доменными данными.
4.  **Ясность**: `workspace/` — где работаю сейчас; `namespaces/` — где лежат знания; `archive/` — где история.
5.  **Контракт**: `manifest.yaml` явно говорит: «Я — `vault/` v2.0, парси меня по `schema.yaml`».

### Миграция с текущей структуры

1.  Создать `_system/manifest.yaml` с `vault_schema_version: 2.0`.
2.  Переместить `meta/registry.yaml` → `_system/schema.yaml`, адаптировать формат.
3.  Перенести `context/`, `meta/` (кроме registry), `delivery/` → в соответствующие `namespaces/`:
    - `context/principles/` → `namespaces/engineering/architecture/`
    - `meta/decisions/` → `namespaces/engineering/decisions/`
    - `delivery/` → `namespaces/engineering/delivery/`
4.  Переместить `funnel/` → `workspace/inbox/`.
5.  Оставить `archive/` как логическое пространство (файлы не двигать, обновить индекс).
6.  Обновить все `id` артефактов, если они включали старые пути (опционально, можно оставить старые `id` для обратной совместимости).

## Критические риски нового подхода

| Риск | Влияние | Митигация |
|------|---------|-----------|
| **Сложность миграции** | Существующие ссылки `[[id]]` могут сломаться при переименовании путей | Поддержать обратную совместимость `id` или предоставить скрипт миграции ссылок |
| **Когнитивная нагрузка** | Новая структура требует переобучения команды | Создать `shared/templates/vault_migration_guide.md` и провести воркшоп |
| **Избыточность для малых проектов** | Не всем нужны `namespaces/` и `_system/` | Разрешить «упрощённый режим»: один `namespace: default`, минимум конфигов |
| **Версионирование схемы** | Изменение `_system/schema.yaml` может сломать старые артефакты | Использовать семвер для `vault_schema_version`, предоставлять миграционные скрипты |

---

## Вердикт

Текущая структура `vault/` **оптимальна для текущего проекта**, но **не портативна** как независимый ресурс.

**Рекомендация**: Принять предложенную универсальную структуру (v2) как целевую. Реализовать миграцию поэтапно:
1.  Сначала добавить `_system/manifest.yaml` и `schema.yaml` (не ломая текущее).
2.  Затем создать `namespaces/` и переносить артефакты по мере их изменения.
3.  В конце — очистить корень от старых папок.

Это обеспечит плавный переход без потери трассируемости.
