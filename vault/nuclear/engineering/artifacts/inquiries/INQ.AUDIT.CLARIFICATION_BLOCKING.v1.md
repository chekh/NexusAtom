## Вопрос 1: Где спецификация ядра?

### Контекст
Методология описывает:
- «Парсер `vault/` → индекс»
- «Резолвер ссылок `[[id]]`»
- «Валидатор против `registry.yaml`»
- «CLI: `list`, `trace`, `create`, `validate`»

### Неопределённость
- Это планы на будущее или уже реализованный код?
- Если реализован — где репозиторий / директория с кодом?
- Если планы — есть ли спецификация интерфейсов (как CLI вызывает парсер, как валидатор читает реестр)?

### Почему блокирует
Без ядра:
- Агенты не могут безопасно работать с `vault/` (нарушение `principle/agent_protocol`)
- Валидация остаётся ручной → ошибки, противоречия
- Методология не может быть протестирована в реальных условиях

### Требуемый ответ
```
[ ] Ядро реализовано: путь к коду / документация по использованию
[ ] Ядро в разработке: ссылка на спецификацию / роадмап / ветку
[ ] Ядро планируется: приоритет, сроки, ответственный
[ ] Ядро не планируется: альтернативный механизм валидации/навигации
```

## Вопрос 2: Как определяется «критичность» типа для агентов?

### Контекст
> `governance/agent_workflow.md`: «Если тип критический (adr, governance, architecture) → требует человеческого аппрува»

### Неопределённость
- Где список критических типов? В `registry.yaml`? В отдельном конфиге?
- Может ли агент предлагать изменения в этот список?
- Что если тип не в списке, но изменение влияет на критический аспект (например, `concept` меняет принцип)?

### Почему блокирует
Без чётких правил:
- Агент либо слишком ограничен (теряет автономность)
- Либо слишком свободен (риск поломки графа)
- Невозможно автоматизировать эскалацию

### Требуемый ответ
```
[ ] Список критических типов: путь к файлу / схема в registry.yaml
[ ] Правила определения критичности: алгоритм / эвристика
[ ] Механизм обновления списка: кто, как, когда
[ ] Обработка пограничных случаев: пример + решение
```

## Вопрос 3: Что происходит при конфликте манифестов?

### Контекст
> `principle/git_integration`: «Манифест-трекинг: каждый коммит привязан к `atom_id` и `manifest.yaml`. CI проверяет целостность хэшей.»

### Неопределённость
- Если два агента одновременно меняют файлы, указанные в одном `manifest.yaml` — как детектируется конфликт?
- Если `content_hash` не совпадает — это ошибка или предупреждение?
- Кто разрешает конфликт: первый коммит, последний, человек, алгоритм?

### Почему блокирует
Без стратегии разрешения конфликтов:
- Параллельная работа агентов ведёт к потере данных
- Агенты не могут координировать изменения
- Доверие к `manifest.yaml` как к SSOT падает

### Требуемый ответ
```
[ ] Детекция конфликтов: механизм (Git? ядро? внешний сервис?)
[ ] Стратегия разрешения: приоритеты, автоматизация, человеческий арбитраж
[ ] Откат при неудаче: как восстановить консистентность
[ ] Пример сценария: пошаговое описание разрешения типичного конфликта
```

## Вопрос 4: Как работает «динамическая агрегация» обзорных документов?

### Контекст
> `principle/logical_archiving`: «Обзорные документы рендерятся из графа по `{{query: ...}}` шаблонам. Ручной мердж исключён.»

### Неопределённость
- Кто/что выполняет рендеринг `{{query: ...}}`?
- Когда происходит рендеринг: при чтении файла? при коммите? по расписанию?
- Где хранится результат рендеринга: кэш, in-memory, генерируется на лету?

### Почему блокирует
Без понимания механизма:
- Невозможно оценить производительность при масштабировании
- Невозможно спроектировать кэширование
- Агенты не знают, когда их изменения станут видны в обзорах

### Требуемый ответ
```
[ ] Движок рендеринга: название / код / спецификация
[ ] Триггер рендеринга: событие / расписание / on-demand
[ ] Стратегия кэширования: TTL, инвалидация, предрендеринг
[ ] Пример запроса: {{query: type=atom AND status=approved}} → результат
```

## Вопрос 5: Как соотносятся `status` в фронтматтере и `_meta.yaml`?

### Контекст
> `structure_analysis.md`: «Для лёгких узлов фронтматтер в `.md` допустим. Для `delivery`-атомов — отдельный `_meta.yaml` обязателен.»

### Неопределённость
- Если файл имеет и фронтматтер, и `_meta.yaml` — какое поле `status` приоритетно?
- Что если они расходятся? Это ошибка валидации или допустимое состояние?
- Как агент должен читать статус: парсить фронтматтер, читать `_meta.yaml`, использовать CLI?

### Почему блокирует
Два источника истины для одного поля:
- Риск рассинхронизации
- Неопределённость для агентов (какой источник читать)
- Усложнение валидации (проверять согласованность)

### Требуемый ответ
```
[ ] Единый источник истины: фронтматтер ИЛИ _meta.yaml (выбрать один)
[ ] Если оба допустимы: правило приоритета + механизм синхронизации
[ ] Валидация рассинхронизации: ошибка / предупреждение / авто-исправление
[ ] Рекомендация для агентов: какой метод чтения использовать
```
