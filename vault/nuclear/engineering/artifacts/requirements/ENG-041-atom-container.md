# Требование: Атом как контейнер доставки ценности

## Описание
Атом — это не «спека» и не «файл». Это физический контейнер, объединяющий все артефакты, необходимые для реализации одной логической функции.

## Обязательные компоненты атома
```
vault/delivery/{domain}/{atom_id}/
├── _meta.yaml          # Метаданные: id, type, status, version, traces
├── spec.md             # Контракт: требования, сценарии (Gherkin), API
├── adr.md              # Архитектурное решение: почему так, а не иначе
├── test_plan.md        # План тестирования: сценарии, критерии приемки
├── tasks.md            # План реализации: шаги, оценка усилий
└── manifest.yaml       # Привязка к коду: какие файлы в репозитории меняет этот атом
```

## Правила
1. **Самодостаточность**: Все, что нужно для реализации, лежит в одной папке.
2. **Методологическая агностичность**: Атом не привязан к Agile/Scrum. Он материализует намерение из любой процессной модели.
3. **Верифицируемость**: Контракт (`spec.md`) валидируется через Pydantic v2 `strict=True`.
4. **Трассируемость**: `manifest.yaml` содержит `commit_hash` для каждого затронутого файла.

## Критерии приемки
- [ ] Атом можно создать через `nexus atom create --type {type} --id {id}`
- [ ] `_meta.yaml` валидируется схемой из `registry.yaml`
- [ ] `manifest.yaml` проверяет существование путей и корректность хэшей
- [ ] Граф строит связи по `traces.*` полям
- [ ] Атом можно перевести в `status: approved` только при наличии обязательных артефактов

## Приоритет
🔴 Критический. Без этого нет атомарности, нет трассировки, нет детерминированной доставки.

## Зависимости
- `meta/registry.yaml` (определение типа `atom` и профиля `delivery`)
- `principle/vault_ssot` (единое хранилище)
- `principle/logical_archiving` (статусы вместо перемещения)
