# Проблема: OpenSpec как монолит

## Контекст
Анализ репозитория `Fission-AI/OpenSpec` выявил фундаментальные ограничения для параллельной и spec-driven разработки.

## Выявленные проблемы
1. **Атомарность = файл**: Базовая единица — `specs/<domain>/spec.md`. Правка одной фичи затрагивает весь файл.
2. **Нет графа зависимостей**: Поиск делегирован LLM. Агент читает файлы целиком или гадает.
3. **Бинарный архив**: `/opsx:archive` ожидает 100% выполнения. Нет легализации техдолга/бэклога.
4. **Отсутствие планирования**: Нет эпиков, сроков, приоритетов, `blocked_by`. Только локальный `tasks.md`.
5. **SSOT нарушен для тестов/доков**: Фокус на требованиях. Остальное «уходит в архив» или теряется.
6. **Раздутый контекст**: Нет lazy loading. Агент читает полные файлы, кэш не инвалидируется.
7. **Разрыв с оркестратором**: Ручная синхронизация. План живёт отдельно от спеков.

## Последствия
- Параллельная работа = race conditions, конфликты мерджа, устаревший контекст.
- Агент не видит зависимости → генерирует несовместимый код.
- Невыполненные задачи либо игнорируются, либо заставляют «дожимать» в ущерб качеству.

## Решение в NexusAtom
- Атомарные файлы + явные типы (`delivery/context`).
- Граф зависимостей (Neo4j/NetworkX) + impact-analysis.
- FSM с `partial_archive` + автогенерация `backlog/`/`debt/`.
- `roadmap.yaml` + двусторонний вебхук с планировщиками.
- `traces: {tests, docs, arch}` как first-class связи.
- Lazy load по `atom_id`, инвалидация по `content_hash`.

## Статус
✅ Решено архитектурно. Требуется реализация ядра.
