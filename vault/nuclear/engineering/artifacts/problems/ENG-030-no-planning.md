# Проблема: Отсутствие слоя планирования

## Тезис
OpenSpec не имеет функционала для стратегического планирования. Инструмент работает в парадигме **Just-in-Time**, а не **Roadmap-Driven**.

## Gap Analysis
| Функция планирования | Статус в OpenSpec | Последствие |
|---------------------|-------------------|-------------|
| Дорожные карты (Roadmaps) | ❌ Отсутствуют | Нет понятия «версия 2.0» или «квартальная цель». Агент оптимизирует локальную задачу, игнорируя глобальную архитектуру. |
| Эпики и декомпозиция | ❌ Отсутствуют | Нет механизма связать 10 мелких спеков в одну большую фичу. |
| Таймлайны и дедлайны | ❌ Отсутствуют | В спеке нет полей `due_date`, `priority`, `milestone`. Невозможно приоритезировать бэклог. |
| Трекинг прогресса | ⚠️ Бинарный | Статус есть только у изменения: `active` / `archived`. Нет промежуточных статусов. |
| Управление зависимостями | ❌ Отсутствует | Нет графа блокировок. Агент не знает, что «Спека Б» не может быть начата, пока не утверждена «Спека А». |

## Сценарий провала
```
# Шаг 1: Ты говоришь агенту "План: Система трейдинга"
Агент: "Ок, создаю спеку specs/trading/data_collector.md"
# (Где общий план? Агент его "забыл", контекст очищен)

# Шаг 2: Ты говоришь "Теперь делаем модель"
Агент: "Ок, создаю specs/trading/ml_model.md"
# (Агент не видит, что модель зависит от формата данных из Шага 1)

# Шаг 3: Интеграция
# Ты получаешь 3 разрозненных спека. 
# Стыковка не была запланирована → рассинхрон, переделка.
```

## Решение в NexusAtom
- **`roadmap.yaml`** как источник истины для эпиков, сроков, приоритетов.
- **FSM с `blocked_by`**: Атом не может стартовать, если зависимости не `approved`.
- **Двусторонний синк**: Вебхук между `vault/` и Linear/Jira/Beads.
- **Impact analysis**: Перед созданием атома — проверка графа на конфликты.

## Статус
✅ Концептуально решено. Требует реализации `PlannerSyncEvent` и маппинга статусов.
