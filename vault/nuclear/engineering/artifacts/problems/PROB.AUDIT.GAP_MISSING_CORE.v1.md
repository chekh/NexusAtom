# Пробел: Отсутствие ядра (парсер, валидатор, CLI)

## Описание
Методология описывает **что** должно работать, но не предоставляет **как** это реализовать. Ключевые компоненты ядра отсутствуют:

| Компонент | Назначение | Статус | Влияние |
|-----------|------------|--------|---------|
| **Парсер `vault/`** | Индексация файлов, извлечение `id`, `status`, `traces` | ❌ Не реализован | Граф не строится, ссылки `[[id]]` не резолвятся |
| **Валидатор** | Проверка артефактов против `registry.yaml` (FSM, обязательные поля) | ❌ Не реализован | Агенты могут создавать невалидные узлы, ломая целостность |
| **CLI** | `create`, `list`, `trace`, `validate`, `approve` | ❌ Не реализован | Ручная работа с файлами → ошибки, низкая скорость |
| **Резолвер ссылок** | Замена `[[id]]` на актуальный путь + статус + превью | ❌ Не реализован | Навигация требует ручного поиска, трассировка неавтоматизирована |
| **Манифест-трекер** | Проверка `content_hash` в `manifest.yaml` против реальных файлов | ❌ Не реализован | Невозможно детектировать несанкционированные изменения |

## Почему это критично

### 1. Нарушение принципа «агенты не пишут напрямую в ФС»
> `principle/agent_protocol`: «Агенты пишут только через API/CLI. Прямая запись в FS блокируется Git-хуками.»

❌ **Проблема**: Нет CLI → агенты вынуждены писать напрямую → нарушение принципа → риск поломки графа.

### 2. Невозможность автоматической валидации
> `principle/dynamic_registry`: «Система валидирует переходы, а не навязывает термины.»

❌ **Проблема**: Нет валидатора → «система» = человек → субъективность, ошибки, медленная обратная связь.

### 3. Отсутствие трассировки в реальном времени
> `structure_analysis.md`: «Ссылки по `id` остаются рабочими независимо от статуса.»

❌ **Проблема**: Нет резолвера → ссылки `[[id]]` = текст, не функциональность → граф «мёртвый».

## Последствия для методологии

1. **Методология остаётся документацией**, а не работающей системой.
2. **Агенты не могут быть полноценными участниками** — им не через что взаимодействовать с `vault/`.
3. **Масштабирование невозможно** — ручное управление графом не масштабируется на 100+ атомов.
4. **Доверие к системе падает** — без автоматической валидации каждый артефакт требует ручного ревью.

## Предложения по закрытию пробела

### 🔴 Фаза 0: Минимальное ядро (необходимо для продолжения аудита)
```
core/
├── parser/
│   ├── __init__.py
│   ├── file_scanner.py      # Сканирование vault/, извлечение фронтматтера
│   └── graph_builder.py     # Построение DAG по traces.* и [[id]]
├── validator/
│   ├── __init__.py
│   ├── registry_loader.py   # Загрузка meta/registry.yaml
│   ├── fsm_checker.py       # Валидация переходов статусов
│   └── schema_validator.py  # Pydantic-валидация полей
├── cli/
│   ├── __init__.py
│   ├── create.py            # nexus create --type <type> --id <id>
│   ├── list.py              # nexus list --status <status> --type <type>
│   ├── trace.py             # nexus trace <id> --direction <upstream|downstream>
│   └── validate.py          # nexus validate <id|all>
└── resolver/
    ├── __init__.py
    └── link_resolver.py     # Замена [[id]] на путь + метаданные
```

### 🟡 Фаза 1: Интеграция с процессом
- Git-хуки: `pre-commit` → `nexus validate`
- GitHub Actions: CI-чек валидации при пулл-реквесте
- Вебхуки: синхронизация с Jira/Linear

### 🟢 Фаза 2: Расширение
- Графический дашборд (NetworkX + Dash / Neo4j)
- Кэширование индекса для производительности
- Поддержка плагинов для кастомных валидаторов

## Блокирующий статус
🔴 **Этот пробел блокирует дальнейшую реализацию методологии**. Без ядра:
- Агенты не могут безопасно работать с `vault/`
- Валидация остаётся ручной → ошибки, противоречия
- Методология не может быть протестирована в реальных условиях

## Рекомендация аудита
**Приостановить развитие документации** до создания минимального ядра (Фаза 0). 
Дальнейший аудит имеет смысл только после того, как методология станет исполняемой спецификацией.
