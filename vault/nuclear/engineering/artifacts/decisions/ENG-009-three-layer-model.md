# ADR: Принятие трёхуровневой архитектуры NexusAtom (Знания → Процессы → Доставка)

## Контекст
Для обеспечения масштабируемости, портативности и чёткого разделения ответственности требуется явное разделение архитектуры NexusAtom на три логических уровня:
1.  **Ядро знаний (Vault Core)** — универсальная база текстовых знаний, SSOT.
2.  **Управление процессами (Orchestration Layer)** — правила, методологии, маршрутизация.
3.  **Доставка ценности (Delivery Layer)** — систематизация артефактов исполнения.

## Рассмотренные варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Монолитная архитектура** (всё в одной плоскости) | Проще на старте, меньше файлов | Невозможно масштабировать, сложно переиспользовать, смешиваются ответственности |
| **Двухуровневая** (знания + доставка) | Лучше, чем монолит | Процессы остаются «вшитыми» в данные или код, сложно менять методологию |
| **Трёхуровневая** (знания → процессы → доставка) | ✅ Чёткое разделение ответственности, портативность ядра, гибкость процессов, модульность доставки | Требует более тщательного проектирования интерфейсов между уровнями |

## Решение
✅ **Принять трёхуровневую архитектуру как целевую модель NexusAtom**:

### 🟦 Уровень 1: Ядро знаний (Vault Core)
**Назначение:** Портативный, агентно-ориентированный контейнер для централизации, валидации и безопасного доступа к знаниям.

**Компоненты:**
```
vault/
├── _system/              # manifest.yaml, security_policy.yaml, nuclear_registry.yaml
├── context/principles/   # Статические принципы, глоссарий
├── context/memory/       # Изолированная память агентов (short_term, long_term)
├── meta/                 # audit/, indexes/, migrations/
├── workspace/            # inbox/, drafts/, current/
└── archive/              # Логический архив (Dark Zone по умолчанию)
```

**Контракт уровня:**
- Само-описываемость через `manifest.yaml`.
- Каскадная валидация локальных контрактов.
- Индексация на уровне ядра (векторный поиск, граф связей).
- Применение Зон тьмы на уровне индексации.

### 🟨 Уровень 2: Управление процессами (Orchestration Layer)
**Назначение:** Организация *того, как* знания создаются, ревьюятся и перемещаются между состояниями.

**Компоненты:**
```
context/workflows/
├── methodologies/        # Agile, Research, Kanban, Stage-Gate
├── fsm-definitions/      # Машины состояний для типов артефактов
├── routing-rules/        # Маршруты ревью, аппрува, эскалации
└── governance-policies/  # Политики качества, compliance, этики
```

**Контракт уровня:**
- Методологическая агностичность: процессы конфигурируются, не захардкожены.
- FSM + гейты: переходы статусов управляются правилами.
- Управление Зонами тьмы: аудит, обоснование, эскалация.

### 🟥 Уровень 3: Доставка ценности (Delivery Layer)
**Назначение:** Конкретная систематизация артефактов, производящих измеримую ценность.

**Компоненты:**
```
nuclear/{resource_id}/
├── _contract.yaml        # Локальный контракт (типы, права, dark_zones)
├── dependencies.yaml     # Зависимости от других ядерных ресурсов
└── artifacts/{artifact_id}/
    ├── spec.md
    ├── adr.md
    ├── test_plan.md
    └── manifest.yaml
```

**Контракт уровня:**
- Контрактная изоляция: каждый ресурс сам определяет свои артефакты.
- Определение Зон тьмы в `_contract.yaml`.
- Манифест-трекинг: связь спеков с физическими файлами.

## Обоснование

1.  **Портативность ядра (L1):** `vault/` как независимый ресурс знаний может использоваться в любом проекте, независимо от методологии или домена.
2.  **Гибкость процессов (L2):** Смена методологии (Agile → Research) = замена конфигов в `context/workflows/`, не переписывание ядра или данных.
3.  **Модульность доставки (L3):** Ядерные ресурсы копируются как модули. Новый проект = выбор нужных `nuclear/` + регистрация в `_system/nuclear_registry.yaml`.
4.  **Сквозная безопасность:** Зоны тьмы определяются на L3, управляются на L2, применяются на L1 — многоуровневая защита без единой точки отказа.
5.  **Ясность ответственности:** Каждый уровень имеет чёткий контракт. Разработчик ядра не думает о процессах, процесс-инженер не думает о индексации, доставщик не думает о шифровании памяти.

## Последствия

### ✅ Положительные
- **Масштабируемость:** Можно начинать с L1 + один ядерный ресурс, добавлять L2 и новые `nuclear/` по мере роста.
- **Переиспользование:** Ядро знаний и процессы — универсальны. Ядерные ресурсы — модульны.
- **Безопасность:** Многоуровневая валидация Зон тьмы, изоляция памяти, каскадные проверки.
- **Эволюция без разрывов:** Изменение одного уровня не ломает другие (при соблюдении контрактов).

### ⚠️ Отрицательные (и митигация)
| Последствие | Митигация |
|-------------|-----------|
| Сложность проектирования интерфейсов между уровнями | Создать `context/templates/level_contracts.md` с явными точками интеграции |
| Оверхед валидации на трёх уровнях | Инкрементальная валидация + кэширование + триггеры только на затронутые компоненты |
| Когнитивная нагрузка на команду | Создать `context/templates/vault_tour.md` с интерактивными примерами для каждого уровня |
| Риск рассинхрона уровней | Ввести `vault health-check --all-levels` для проверки целостности связей L1↔L2↔L3 |

## План внедрения

### Этап 1: Ядро знаний (L1) — недели 1-3
- [ ] Реализовать парсер `_system/manifest.yaml` и `nuclear_registry.yaml`.
- [ ] Реализовать каскадную валидацию локальных контрактов.
- [ ] Реализовать индексатор с исключением Зон тьмы из векторного поиска.
- [ ] Реорганизовать `context/` для изоляции памяти агентов.

### Этап 2: Доставка ценности (L3) — недели 4-6
- [ ] Реализовать парсер `_contract.yaml` и `dependencies.yaml` ядерных ресурсов.
- [ ] Реализовать манифест-трекинг и валидацию артефактов.
- [ ] Создать `context/templates/nuclear_resource_template/` с примерами.
- [ ] Протестировать copy-paste ядерного ресурса в тестовый проект.

### Этап 3: Управление процессами (L2) — недели 7-9
- [ ] Реализовать парсер `methodologies/` и `fsm-definitions/`.
- [ ] Реализовать маршрутизацию задач и гейты ревью.
- [ ] Интегрировать аудит Зон тьмы и логирование обходов.
- [ ] Создать `context/templates/workflow_starter_kit.md` для типовых методологий.

### Этап 4: Интеграция и документация — неделя 10
- [ ] Протестировать сквозной поток: идея → процесс → доставка.
- [ ] Создать интерактивный `vault tour` для онбординга.
- [ ] Обновить `README.md` с описанием трёхуровневой модели.

## Критерии успеха
- [ ] Новый проект инициализирует `vault/` с нужными уровнями за <10 минут.
- [ ] Копирование ядерного ресурса + регистрация = рабочий модуль за <2 минуты.
- [ ] Попытка агента прочитать `read_blind` зону возвращает `AccessDenied` на уровне индекса.
- [ ] Смена методологии в `context/workflows/` не требует изменения ядра или данных.
- [ ] `vault health-check --all-levels` проходит без ошибок для тестового проекта.

## Статус
✅ **Принято**. Следующий шаг: начало реализации Фазы 1 (Ядро знаний — L1).
