# ADR: Переименование `specs/` → `vault/`

## Контекст
Изначально использовалась папка `specs/`, унаследованная от OpenSpec. Однако в процессе обсуждения стало ясно, что система хранит не только спецификации, а **всё знание проекта**: идеи, заметки, исследования, правила, архитектуру, эпики, атомы доставки.

## Рассмотренные варианты
| Имя | Плюсы | Минусы |
|-----|---------|--------|
| `specs/` | Привычно, кратко | Слишком узко, не отражает полноту хранилища |
| `knowledge/` | Описательно, понятно | Длинное, менее «техническое», конфликт с `docs/` |
| `atoms/` | Подчёркивает атомарную модель | Может путать: атом = тип или контейнер? |
| `nexus/` | Брендингово, соответствует названию | Слишком мета, может конфликтовать с `src/nexus_atom/` |
| `vault/` | ✅ Коротко, метафора «хранилища», работает с реакторной аналогией | Может ассоциироваться с секретами (но контекст прояснит) |

## Решение
✅ **`vault/`** — универсальное, краткое, семантически точное имя для корня хранилища.

## Обоснование
1. **Нейтральность**: Не намекает на один тип артефакта.
2. **Краткость**: Удобно в CLI: `nexus create vault/...`.
3. **Семантика**: Отражает идею «единого хранилища истины».
4. **Совместимость**: Не конфликтует с `src/`, `docs/`, `.github/`.
5. **Брендинг**: Работает с метафорой реактора (`vault` = хранилище «топлива» для `reactor`).

## Последствия
- [ ] Все существующие пути обновлены: `specs/` → `vault/`
- [ ] `.specrc.yaml`: `root: vault/`
- [ ] Документация и примеры обновлены
- [ ] Индексатор сканирует `vault/`, не `specs/`

## Статус
✅ Принято. Реализовано в коммите `docs(vault): initialize core principles`.
