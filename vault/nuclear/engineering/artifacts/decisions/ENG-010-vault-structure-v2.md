# ADR: Принятие универсальной структуры `vault/` v2

## Контекст
Текущая структура `vault/` (`context/`, `meta/`, `delivery/`) эффективна для разработки NexusAtom, но не портативна как независимый ресурс знаний. Для поддержки разнородных проектов и обеспечения масштабируемости требуется пересмотр логики структурирования.

## Рассмотренные варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Оставить как есть** | Просто, не требует миграции | Не портативно, жёстко привязано к домену разработки ПО |
| **Полностью плоская структура** (все артефакты в корне, навигация только по `id`) | Максимальная гибкость, нет папок | Неприемлемо для человеческой навигации, сложно контролировать доступ |
| **Универсальная структура v2** (`_system/`, `namespaces/`, `workspace/`, `archive/`, `shared/`) | Портативна, само-описываема, гибка через `schema.yaml`, сохраняет человеческую навигацию | Требует миграции, добавляет сложность конфигурации |

## Решение
✅ **Принять универсальную структуру `vault/` v2** как целевую архитектуру.

## Обоснование
1.  **Портативность**: Структура адаптируется под любой домен через конфигурацию `namespaces/` в `schema.yaml`, не требуя изменения кода ядра.
2.  **Само-описываемость**: `manifest.yaml` явно определяет версию схемы и правила ресурса, позволяя инструментам автоматически парсить и валидировать.
3.  **Изоляция ответственности**: `_system/` для системных конфигов, `namespaces/` для доменных знаний, `workspace/` для активной работы — чёткие границы.
4.  **Масштабируемость**: Поддержка инкрементальной индексации, шардирования по `namespaces/`, ленивой загрузки.
5.  **Безопасность**: Явная ACL в `schema.yaml`, аудит изменений `_system/`, валидация при импорте.

## Последствия

### ✅ Положительные
- Возможность использовать один и тот же `vault/` скелет для разных проектов (софт, исследование, бизнес).
- Упрощение миграции между версиями методологии через `schema.yaml` и скрипты.
- Улучшенная навигация: `workspace/` для текущего, `namespaces/` для организованного, `archive/` для истории.
- Чёткий контракт на импорт/экспорт знаний.

### ⚠️ Отрицательные (и митигация)
| Последствие | Митигация |
|-------------|-----------|
| Сложность миграции с v1 на v2 | Предоставить скрипт `vault migrate v1→v2` с бэкапом и валидацией |
| Когнитивная нагрузка от гибкости `namespaces/` | Создать `shared/templates/namespace_starter_kit.md` с примерами для типовых доменов |
| Необходимость обновлять ядро для поддержки `schema.yaml` | Реализовать поддержку постепенно: сначала `manifest.yaml`, затем `schema.yaml` |

## План внедрения

### Этап 1: Подготовка (неделя 1)
- [ ] Обновить `MET.ANALYSIS.UNIVERSAL_SKELETON.v1` с учётом рекомендаций экспертной панели → `v2`.
- [ ] Создать `vault/_system/manifest.yaml` и `vault/_system/schema.yaml` на основе текущего `meta/registry.yaml`.
- [ ] Протестировать валидацию: `vault validate --all` должен работать с новой схемой.

### Этап 2: Миграция (неделя 2-3)
- [ ] Запустить скрипт `vault migrate v1→v2` для переименования путей и обновления метаданных.
- [ ] Перенести артефакты в соответствующие `namespaces/` (например, `context/principles/` → `namespaces/engineering/architecture/`).
- [ ] Обновить все внутренние ссылки `[[id]]`, если они использовали старые пути (опционально, если `id` не менялись).

### Этап 3: Интеграция (неделя 4)
- [ ] Обновить CLI: добавить поддержку `--namespace` в `search`, `list`, `create`.
- [ ] Реализовать `vault import` и `vault export` с валидацией против `schema.yaml`.
- [ ] Настроить аудит-лог для изменений в `_system/`.

### Этап 4: Документация и онбординг (неделя 5)
- [ ] Создать `shared/templates/vault_migration_guide.md` для пользователей.
- [ ] Провести воркшоп по новой структуре для команды.
- [ ] Обновить `README.md` проекта с описанием универсальной структуры.

## Критерии успеха
- [ ] Новый проект может инициализировать `vault/` за <5 минут, настроив только `namespaces/` в `schema.yaml`.
- [ ] Миграция существующего `vault/` v1 на v2 проходит без потери трассируемости (`id`, `traces`).
- [ ] CLI `vault search --namespace engineering` возвращает результаты только из этого пространства.
- [ ] Попытка импортировать невалидный артефакт отклоняется с понятной ошибкой.

## Статус
✅ **Принято**. Следующий шаг: обновление спецификации и начало реализации Фазы 1.
