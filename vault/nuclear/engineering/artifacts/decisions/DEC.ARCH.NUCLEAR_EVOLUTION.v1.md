# ADR: Принятие эволюции ядерных ресурсов (Локальные контракты, Память, Зоны тьмы)

## Контекст
Для обеспечения истинной портативности ядерных ресурсов, эффективной работы агентов и экономии вычислительных ресурсов требуется:
1.  Перенести контракты ядерных ресурсов внутрь самих ресурсов (`_contract.yaml`).
2.  Явно отделить память агентов от принципов проекта.
3.  Ввести механизм «Зон тьмы» для ограничения доступа агентов к нерелевантным или чувствительным данным.

## Рассмотренные варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Централизованные контракты** (как было) | Проще валидировать, единая точка контроля | Низкая портативность, сложно копировать ресурсы |
| **Полностью децентрализованные контракты** | Максимальная гибкость, каждый ресурс сам по себе | Риск противоречий, сложно обеспечить глобальную безопасность |
| **Гибридная модель** (локальные контракты + глобальная политика безопасности) | ✅ Баланс: портативность + контроль, каскадная валидация | Требует реализации каскадной валидации, чуть сложнее онбординг |

## Решение
✅ **Принять гибридную модель эволюции ядерных ресурсов**:

### 1. Локальные контракты с каскадной валидацией
- `_contract.yaml` живёт внутри `nuclear/{resource_id}/`.
- При регистрации ресурса в `_system/nuclear_registry.yaml` ядро проверяет:
  - Соответствие `_contract.yaml` глобальной `_system/security_policy.yaml`.
  - Наличие и валидность зависимостей (`dependencies.yaml`).
- Конфликт → ошибка `ContractViolation`, ресурс не активируется.

### 2. Изоляция памяти агентов
```
context/
├── principles/         # Статика: принципы, ценности (не меняется агентами)
├── glossary/           # Статика: термины
└── memory/             # Динамика: память агентов
    └── agents/{agent_id}/
        ├── short_term/      # Сессии (очищаются/архивируются)
        ├── long_term/       # Усвоенные знания (версионируются)
        └── _meta.yaml       # Метаданные: роль, права, история
```
- Память шифруется отдельно (настраивается в `acl.yaml`).
- При экспорте проекта память агентов может быть исключена (флаг `--exclude-agent-memory`).

### 3. Зоны тьмы (Dark Zones) на уровне индексации
- Определяются в `_contract.yaml` ресурса или глобально в `_system/security_policy.yaml`.
- Типы:
  - `read_blind`: Агент не видит контент (возвращается `AccessDenied`).
  - `write_prohibited`: Агент видит, но не может писать.
  - `token_economic_block`: Предупреждение о высокой стоимости токенов.
- **Критическое правило**: Зоны тьмы применяются **на уровне индексации**. Запрещённые пути не попадают в векторный индекс и не возвращаются в результатах поиска.

### 4. Обоснование Зон тьмы
- Каждая `read_blind` зона должна иметь явное поле `reason` в контракте:
  ```yaml
  dark_zones:
    - path: "nuclear/legal/contracts/"
      type: read_blind
      reason: "Содержит персональные данные, защищённые законом о приватности."
      requires_audit_log: true
  ```
- Аудит проверяет обоснованность причин при изменении контракта.

## Обоснование (Вердикт Мастера)

1.  **Портативность без потери контроля**: Локальные контракты позволяют копировать ядерные ресурсы как модули. Каскадная валидация гарантирует, что модуль не нарушит глобальные правила безопасности нового проекта.
2.  **Экономия и фокус для агентов**: Зоны тьмы на уровне индексации предотвращают трату токенов на нерелевантные данные и снижают риск галлюцинаций.
3.  **Гигиена и безопасность памяти**: Изоляция `context/memory/agents/` позволяет управлять жизненным циклом памяти отдельно от принципов проекта, шифровать её и экспортировать выборочно.
4.  **Этика и прозрачность**: Требование `reason` для Зон тьмы предотвращает злоупотребления и обеспечивает аудируемость решений.

## Последствия

### ✅ Положительные
- **Модульность**: Ядерные ресурсы становятся переиспользуемыми модулями. Скопировал папку + зарегистрировал — готово.
- **Экономия**: Зоны тьмы экономят токены, ускоряют поиск, снижают риск утечек.
- **Безопасность**: Каскадная валидация + шифрование памяти + аудит Зон тьмы = многоуровневая защита.
- **Ясность**: Изоляция памяти упрощает онбординг новых агентов и отладку.

### ⚠️ Отрицательные (и митигация)
| Последствие | Митигация |
|-------------|-----------|
| Сложность каскадной валидации | Реализовать кэширование результатов валидации (`meta/indexes/contract_cache.sqlite`) |
| Риск фрагментации контрактов | Создать `context/templates/nuclear_resource_template/` с лучшими практиками |
| Когнитивная нагрузка от Зон тьмы | Добавить команду `vault dark-zones --list` для явного показа активных ограничений |
| Оверхед индексации с исключениями | Реализовать ленивую индексацию: запрещённые пути просто пропускаются при сканировании |

## Критерии успеха
- [ ] Копирование `nuclear/engineering/` в новый проект + регистрация в `_system/nuclear_registry.yaml` = рабочий ресурс за <2 минуты.
- [ ] Попытка агента прочитать `read_blind` зону возвращает `AccessDenied`, а не контент.
- [ ] Векторный поиск не возвращает результаты из Зон тьмы.
- [ ] Память агентов шифруется (если включено) и не смешивается с `principles/`.
- [ ] Аудит логирует попытки доступа к Зонам тьмы с указанием `reason`.
