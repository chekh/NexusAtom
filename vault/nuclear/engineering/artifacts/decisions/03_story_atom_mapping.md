# ADR: Story (Agile) = Atom (NexusAtom) по смыслу, но не по форме

## Контекст
В процессе обсуждения возник вопрос: если `Story = Atom` по ценности, зачем вводить новый термин? Не избыточно ли это?

## Анализ различий
| Критерий | Story (в Jira/Linear) | Atom (в NexusAtom) |
|----------|----------------------|-------------------|
| **Форма** | Карточка с текстом, чек-листом, комментариями | Физическая папка с версионируемыми файлами (`spec.md`, `adr.md`, `manifest.yaml`) |
| **Валидация** | Человеческая: «понятно ли ТЗ?» | Машинная: `pydantic.validate()`, `manifest.check_hashes()`, `fsm.validate_transition()` |
| **Контекст** | Размазан: ссылка на Confluence, скрин в комментарии, код в другом репо | Изолирован: всё, что нужно для реализации, лежит в одной папке |
| **Трассируемость** | Слабая: `Story → PR` (через описание, не всегда актуально) | Детерминированная: `Atom → manifest.yaml → [file@commit_hash]` |
| **Потребитель** | Человек (разработчик, тестировщик, ПО) | Человек + Агент (LLM/автогенератор кода) |

## Решение
✅ **Story = Atom по смыслу, но Atom — это «исполняемая спецификация»**.

## Обоснование
1. **Story** отвечает на `ЧТО` и `ЗАЧЕМ` (бизнес-ценность).
2. **Atom** отвечает на `КАК`, `ЧЕМ` и `ГДЕ ЗАФИКСИРОВАНО` (детерминированная доставка).
3. Они не конкурируют, а дополняют друг друга:
   - Продукт создаёт Story в Jira → NexusAtom автоматически создаёт атом с тем же `external_refs`.
   - Разработчик/агент работает только с папкой атома.
   - При завершении атома статус в Jira обновляется автоматически.

## Последствия
- [ ] `_meta.yaml` атома содержит `external_refs: {jira: "AUTH-123"}`
- [ ] Вебхук синхронизирует статусы между Jira/Linear и `vault/`
- [ ] CLI: `nexus create atom --from-jira AUTH-123` генерирует папку атома из тикета

## Статус
✅ Принято. Зафиксировано в `principle/atom_container.md`.
