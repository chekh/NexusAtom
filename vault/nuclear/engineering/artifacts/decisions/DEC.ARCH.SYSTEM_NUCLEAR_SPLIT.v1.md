# ADR: Принятие двухуровневой модели ресурсов `vault/` (System vs Nuclear)

## Контекст
Для обеспечения адаптивности `vault/` под любые проекты (от личной записной книжки до энтерпрайз-разработки) требуется чёткое разделение на **обязательные системные ресурсы** и **динамические ядерные ресурсы**. Текущая структура (`context/`, `meta/`, `delivery/`) не обеспечивает достаточной гибкости и портативности.

## Рассмотренные варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Оставить текущую структуру** | Просто, не требует миграции | Не адаптивна: жёсткие папки не подходят для нетипичных проектов |
| **Полностью плоская структура** (все артефакты в корне, навигация только по `id`) | Максимальная гибкость | Неприемлемо для человеческой навигации, сложно контролировать доступ, нет изоляции системного/доменного |
| **Двухуровневая модель** (системные ресурсы + ядерные ресурсы) | ✅ Баланс стандартизации и гибкости, портативность, контрактная гарантия, агентская память как первый класс | Требует миграции, добавляет сложность конфигурации |

## Решение
✅ **Принять двухуровневую модель ресурсов `vault/`** как целевую архитектуру:
```
vault/
├── _system/              # Контракты, схема, манифест (обязательно)
├── context/              # Принципы, агентская память, общие знания (обязательно)
├── meta/                 # Аудит, индексы, миграции (обязательно)
├── nuclear/              # Ядерные ресурсы: engineering, research, business... (динамически)
├── workspace/            # Активная работа: inbox, drafts, current (обязательно)
└── archive/              # Логический архив (обязательно)
```

## Обоснование
1.  **Портативность**: Системные ресурсы (`_system/`, `context/`, `meta/`) одинаковы для любого проекта. Ядерные ресурсы (`nuclear/`) конфигурируются через `schema.yaml`, позволяя адаптировать `vault/` под любой домен без изменения кода ядра.
2.  **Агентская память как первый класс**: Структура `context/agents/{id}/` делает память версионируемой, экспортируемой, трассируемой. Интеграция с RAG и управление контекстным окном становятся архитектурными примитивами.
3.  **Контрактная гарантия**: Каждый ядерный ресурс имеет явный контракт в `schema.yaml`. Инструменты могут валидировать соответствие до записи, предотвращая дрейф структуры.
4.  **Адаптивность без разрывов**: Новый проект начинается с `profile: minimal` (один ядерный ресурс, базовые типы). При росте добавляются новые ядерные ресурсы и типы через обновление `schema.yaml`, не ломая старые артефакты.
5.  **Изоляция ответственности**: Системные конфиги (`_system/`) не смешиваются с доменными данными (`nuclear/`). Упрощает поддержку, аудит, эволюцию.

## Последствия

### ✅ Положительные
- Возможность использовать один и тот же `vault/` скелет для любых проектов: записная книжка, исследование, software dev, юридический документ.
- Упрощение миграции между версиями методологии через `schema.yaml` и скрипты.
- Улучшенная навигация: `workspace/` для текущего, `nuclear/` для организованного, `archive/` для истории.
- Чёткий контракт на импорт/экспорт знаний и агентской памяти.
- Безопасность: шифрование памяти агентов, аудит изменений системных ресурсов, валидация контрактов.

### ⚠️ Отрицательные (и митигация)
| Последствие | Митигация |
|-------------|-----------|
| Сложность миграции с текущей структуры | Предоставить скрипт `vault migrate to-system-nuclear` с бэкапом, валидацией и опцией `--preserve-ids` |
| Когнитивная нагрузка от двухуровневой структуры | Создать `context/templates/vault_starter_guide.md` с примерами для разных типов проектов + интерактивный `vault tour` |
| Необходимость обновлять ядро для поддержки валидации контрактов | Реализовать поддержку постепенно: сначала `_system/manifest.yaml`, затем `schema.yaml`, затем валидатор ядерных ресурсов |
| Риск over-engineering для малых проектов | Добавить `profile: minimal` в `manifest.yaml` с упрощённой `schema.yaml` и минимумом обязательных полей |

## План внедрения

### Этап 1: Подготовка (неделя 1)
- [ ] Обновить `MET.ANALYSIS.SYSTEM_NUCLEAR_MODEL.v1` с учётом рекомендаций экспертной панели → `v2`.
- [ ] Создать `vault/_system/manifest.yaml` и `vault/_system/schema.yaml` на основе текущего `meta/registry.yaml`.
- [ ] Реорганизовать `context/` для поддержки агентской памяти: создать `context/agents/` с шаблоном `_meta.yaml`.
- [ ] Протестировать валидацию: `vault validate --all` должен работать с новой схемой.

### Этап 2: Миграция (неделя 2-3)
- [ ] Запустить скрипт `vault migrate to-system-nuclear` для:
  - Переименования `delivery/` → `nuclear/engineering/` (для текущего проекта).
  - Перемещения `funnel/` → `workspace/inbox/`.
  - Создания `_system/`, `meta/`, `archive/` с базовыми файлами.
- [ ] Обновить все внутренние ссылки `[[id]]`, если они использовали старые пути (опционально, если `id` не менялись).
- [ ] Протестировать целостность: `vault validate --all` после миграции.

### Этап 3: Интеграция (неделя 4)
- [ ] Обновить CLI: добавить поддержку `--nuclear` в `search`, `list`, `create`.
- [ ] Реализовать `vault import` и `vault export` с валидацией против `schema.yaml`.
- [ ] Настроить аудит-лог для изменений в `_system/` и доступа к `context/agents/`.
- [ ] Реализовать базовую валидацию контрактов ядерных ресурсов при коммите.

### Этап 4: Документация и онбординг (неделя 5)
- [ ] Создать `context/templates/vault_starter_guide.md` с примерами для `profile: minimal`, `standard`, `enterprise`.
- [ ] Создать `context/templates/agent_memory_guide.md` с примерами структурирования памяти.
- [ ] Провести воркшоп по новой структуре для команды.
- [ ] Обновить `README.md` проекта с описанием двухуровневой модели.

## Критерии успеха
- [ ] Новый проект может инициализировать `vault/` за <5 минут, настроив только `profile` и `nuclear/` в `schema.yaml`.
- [ ] Миграция существующего `vault/` на новую модель проходит без потери трассируемости (`id`, `traces`).
- [ ] CLI `vault search --nuclear engineering` возвращает результаты только из этого ресурса.
- [ ] Попытка записать артефакт, не соответствующий контракту ядерного ресурса, отклоняется с понятной ошибкой.
- [ ] Агентская память шифруется (если включено в `acl.yaml`) и очищается при завершении сессии (если настроено).
